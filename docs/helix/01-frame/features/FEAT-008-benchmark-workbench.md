# Feature Specification: FEAT-008 - Benchmark Workbench

**Feature ID**: FEAT-008
**Status**: Accepted
**Priority**: P0
**Owner**: Fizeau Team

> **Update 2026-05-16**: ADR-016 ("Cells Are Self-Describing Evidence")
> replaces the historical `lane:`/`recipe:` data-model blocks with profiles
> as the sole authoring unit. Cells embed the full resolved profile
> snapshot at write time, so the workbench reads profile metadata directly
> from each cell's `report.json` instead of joining back to a separate
> catalog. Surviving `lane*` field references below refer to back-compat
> labels carried in legacy cells (filterable / hideable), not to any live
> lane block.

## Overview

The benchmark microsite needs a raw analytical surface, not another
curated report table. Operators need to answer questions across every
observed benchmark cell: which model/runtime/GPU combinations pass a
given test, which combinations pass under a VRAM ceiling, what failures
cluster by harness or provider, and what token/cost/runtime tradeoffs
exist for a selected subset.

FEAT-008 defines the `/benchmarks/explorer/` workbench. It exposes the
full benchmark cell datatable generated by
`scripts/website/build-benchmark-data.py`, keeps all collected columns
available, and adds browser-side search, sort, filter, grouping, and
aggregate views without a server component.

The explorer is organized as one route-addressable workbench with a
persistent local navigation rail. Its panes are Summary, Raw database,
Comparison, and Combinations. Pane selection must be deep-linkable from
the URL so users can share the analytical surface they are discussing
without losing the static-site deployment model.

## Problem Statement

The current benchmark pages are optimized for narrative slices and small
summary tables. They cannot support thousands of raw cells with hundreds
of collected variables. The existing table UI also caps useful width and
forces operators into horizontal scrolling without analytical controls.

The first-class unit of analysis is a benchmark cell: one suite, task,
lane, and rep with pass/fail, process outcome, tokens, cost, timing,
hardware, runtime, quantization, harness, provider, sampling, and
provenance fields. Every collected field must remain queryable even
when the default visible column set is smaller.

## Requirements

### Functional Requirements

1. The workbench loads the generated Parquet benchmark cell artifact
   (`website/static/data/cells.parquet`) directly in the browser.
2. The generated cell artifact contains only result-bearing attempts:
   explicit timeouts, completed graded passes, and completed graded
   failures. Invalid setup/provider/auth rows and other ungraded runs
   are excluded from the published table and counted in the manifest.
3. The workbench exposes all collected columns for those valid cells through an
   interactive grid column chooser, including internal/debug fields.
4. The default visible columns emphasize operator questions:
   suite, task, subsets, result state, pass/fail, final status,
   harness, provider, model, model quant, K/V quant, KV cache, engine,
   MTP, GPU, GPU RAM, machine, rep, tokens, cost, wall time, and
   timestamps.
5. Users can search text across the important descriptor fields without
   knowing the exact column name.
6. Users can filter by result state, selected task, model, engine, GPU,
   pass-only, maximum GPU RAM, and low/medium-cardinality enum fields
   such as model family, quantization, provider, harness, lane,
   deployment class, hardware family, and sampling settings. Users can
   continue filtering, sorting, and grouping in the grid itself.
7. Saved views answer the currently required questions:
   all cells; passing cells for the selected test; passing cells for the
   selected test on the selected GPU; passing cells for the selected
   test under a selected GPU RAM ceiling; recent failures.
8. Aggregate readouts update automatically for the current filtered
   set: row count, pass count/rate, timeout count, distinct models,
   distinct GPUs, token total, known cost total, and p50 wall time.
9. A combination aggregate table groups by test, model, quantization,
   KV cache, engine, MTP, GPU, and GPU RAM, then reports rows, passes,
   failures, timeouts, pass rate, cost, tokens, and wall p50.
10. A pairwise gap table compares two model families and groups pass-rate
    deltas by high-signal dimensions: task category, difficulty, task,
    outcome, engine, model quant, deployment class, GPU, provider, and
    harness. This supports questions like "where is Qwen trailing
    Claude/Anthropic, and by what kind of task or runtime condition?"
11. The workbench must not require a server, database daemon, or private
   API. Static hosting is sufficient.
12. Terminal-Bench task slugs in the grid link to the canonical task
    detail page for that task.
13. The Summary pane describes the result-bearing dataset and renders
    summary statistics plus distribution/time charts for task type,
    GPU type, wall time by GPU, and wall time by model.
14. The Raw database pane owns the raw search/sort/filter controls and
    lets the datagrid take the pane's available width and height.
15. The Comparison pane owns the pairwise comparison controls and scope
    filters; its table headers are clickable sortable controls.
16. The Combinations pane owns the combination aggregate controls; its
    table headers are clickable sortable controls.
17. The left-hand explorer navigation remains stable while users move
    among panes.
18. The page must honor `website/DESIGN.md`: scientific instrument
    aesthetic, no shadows, design tokens, dense tables, stable sizing,
    and no marketing layout.

### Data Requirements

The cell datatable must include, at minimum:

- Identity and provenance: row id, source report, suite, task,
  task subsets, profile, lane id, descriptor, timestamps.
- Harness/provider/runtime: harness, harness label, provider type,
  provider surface, provider model, engine, engine version, backend,
  runtime props, max context, MTP flag.
- Model and quantization: model, model id, display name, family,
  profile snapshot, quant display, model quant, weight bits, KV cache,
  K quant, V quant, KV disk state.
- Hardware: machine, machine label, hardware profile, chip, chip
  family, GPU vendor/model/RAM, VRAM, system memory, memory type, TDP,
  CPU, OS.
- Outcome: result state, process outcome, grading outcome, reward,
  grader-passed, passed,
  final status, invalid class, terminated-mid-work, LLM request state,
  exit code.
- Resource use: turns, tool calls/errors, input/output/cached/retried
  tokens, reasoning tokens, total tokens, wall seconds, cost USD.
- Sampling and benchmark metadata: reasoning, temperature, top-p,
  top-k, max output tokens, context tokens, dataset, dataset version,
  Fizeau tools version.

Additional collected fields beyond this list remain available; the
workbench may default-hide them but must not drop them from the loaded
table.

## Non-Functional Requirements

- **Scale**: Initial target is thousands of rows and hundreds of
  columns. The design must tolerate growth beyond 10k rows without
  replacing the page architecture.
- **Performance**: Initial load should stream from compressed Parquet
  and avoid loading the 70MB JSON artifact for normal browsing.
- **Portability**: Works as a static microsite page under Hugo/Hextra.
- **Shareability**: Saved views are deterministic enough that story
  pages can link to the workbench with query parameters in a follow-up.
- **Accessibility**: Native form controls keep labels visible, and
  workbench status changes are announced in a live region.

## Success Metrics

- An operator can answer "which combinations pass this test" from the
  explorer without editing code.
- An operator can answer "which models passed this test on RTX 3090"
  and "which models passed this test with less than 24GB VRAM" from the
  explorer.
- An operator can answer "where is the pass-rate gap between Qwen and
  Anthropic/Claude largest" by task category, difficulty, individual
  task, runtime, quantization, provider, and hardware dimension.
- A reader can deep-link directly to Summary, Raw database, Comparison,
  or Combinations without a server route.
- All generated cell columns are available in the grid.
- The workbench builds with the microsite and does not regress existing
  benchmark report pages.
- The implementation has a clear no-server analytical architecture
  captured in an ADR.

## Out of Scope

- Authentication or private result publication.
- Live benchmark ingestion.
- Replacing the curated story pages.
- Server-side Datasette, SQLite, or DuckDB services.
- Editing benchmark records in the browser.
