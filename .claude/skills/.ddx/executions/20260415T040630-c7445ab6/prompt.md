    <title>exec: execute-loop --project flag to target a specific project when server is multi-project</title>
    <description>
&#39;ddx agent execute-loop&#39; help says &#39;execute-loop scans the current project&#39;s execution-ready bead queue.&#39; When submitting to a multi-project server, there is no way to say &#39;run this loop against project X&#39;. The server picks whichever project is the current server context, ignoring the caller&#39;s CWD.&#xA;&#xA;## Observed&#xA;&#xA;From CWD /Users/erik/Projects/axon:&#xA;&#xA;  $ ddx agent execute-loop --once --harness claude --model claude-sonnet-4-6 --json&#xA;  {&#xA;    &#34;project_root&#34;: &#34;/Users/erik/Projects/ddx&#34;,  # WRONG&#xA;    &#34;state&#34;: &#34;running&#34;,&#xA;    &#34;worker_id&#34;: &#34;worker-20260415T011331-d0a4&#34;&#xA;  }&#xA;&#xA;I wanted the worker to pick an axon bead. The server ran the worker against the ddx repo instead. There is no --project flag, no env var, nothing in the CLI to redirect the server&#39;s project scope.&#xA;&#xA;Currently the only way to target a specific project with execute-loop is --local mode (which spawns a subprocess using the caller&#39;s CWD as the project root). This defeats the purpose of having a server in a multi-project deployment.&#xA;&#xA;## Scope&#xA;&#xA;- --project &lt;path&gt; or --project &lt;name&gt; flag on execute-loop, execute-bead, and &#39;ddx agent run&#39; that tells the server which project_root to use&#xA;- Env var fallback: DDX_PROJECT_ROOT or similar&#xA;- Default to CWD if unset and no flag&#xA;- Server refuses ambiguous/unmatched project names with a clean error&#xA;- Integration test: from CWD A, submit a loop targeting project B; assert worker runs in B&#39;s project_root&#xA;&#xA;## Why this matters for the user&#39;s workflow&#xA;&#xA;Without this, the ONLY way to drive multi-project execution from one shell is --local, which (a) loses the server&#39;s queue visibility, (b) ties the parent process to the child, and (c) negates the &#39;ddx agent execute-loop submits to server by default&#39; ergonomic that execute-loop&#39;s --help promises.&#xA;&#xA;## Acceptance&#xA;&#xA;- [ ] --project flag wired through execute-loop/execute-bead/run&#xA;- [ ] Env var fallback documented&#xA;- [ ] Default behavior: CWD-based project detection&#xA;- [ ] Integration test for cross-project submission&#xA;- [ ] execute-loop --help updated to describe project targeting
