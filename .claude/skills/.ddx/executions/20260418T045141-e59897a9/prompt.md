    <title>execute-bead: propagate caller context through Runner.Run and RunAgent</title>
    <description>
## Root cause RC1 from ddx-0a651925&#xA;&#xA;`cli/internal/agent/runner.go:239` (Runner.Run) and `cli/internal/agent/agent_runner.go:163` (Runner.RunAgent) both call `context.WithCancel(context.Background())`, discarding any context from the caller. `RunOptions` exposes no ctx field. `agent.ExecuteBead` likewise takes no ctx, so `server.WorkerManager.runWorker`&#39;s ctx never reaches the agent call.&#xA;&#xA;As a result, `WorkerManager.Stop(id)` (server/workers.go:648) cancels only the worker goroutine; the embedded `agentlib.Run` call (and any subprocess it launched) keeps running until SIGKILL. This is the structural defect behind 33–142h hung workers.&#xA;&#xA;## Scope&#xA;&#xA;1. Add `Context context.Context` to `RunOptions` (or change signatures to `Run(ctx, opts)`). Default to `context.Background()` when nil.&#xA;2. Derive the internal ctx from the caller&#39;s rather than `context.Background()`.&#xA;3. Thread ctx through `agent.ExecuteBead` → `runner.Run` → `agentlib.Run` → provider HTTP.&#xA;4. Update `server.WorkerManager.runWorker`&#39;s `singleTierAttempt` closure to pass `ctx` into `agent.ExecuteBead`.&#xA;&#xA;## Non-goals&#xA;&#xA;- New wall-clock timeout logic (covered by sibling RC2 bead)&#xA;- Operator-facing stop command (covered by ddx-b808df39)
    </description>
    <acceptance>
- [ ] `RunOptions` accepts a caller ctx, or `Runner.Run`/`Runner.RunAgent` accept ctx explicitly&#xA;- [ ] The internal cancel is derived from the caller ctx (no `context.Background()` seeds inside Run/RunAgent)&#xA;- [ ] `agent.ExecuteBead` takes ctx and passes it down&#xA;- [ ] `server.WorkerManager.Stop(id)` cancels the running agent provider call within 2s in a unit test (fake provider blocks on ctx)&#xA;- [ ] Regression test: runner with a canceled ctx returns promptly (&lt;500ms) regardless of mocked provider latency
    </acceptance>
