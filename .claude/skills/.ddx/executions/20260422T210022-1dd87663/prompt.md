    <title>Workers screen: clarify purpose vs Sessions and add lifecycle controls</title>
    <description>
## Observed&#xA;&#xA;`/nodes/.../projects/.../workers` renders a table of workers but has no buttons, no controls, and no explanation of how it differs from `/sessions`. The layout (`cli/internal/server/frontend/src/routes/nodes/[nodeId]/projects/[projectId]/workers/+layout.svelte`) shows ID / Kind / State / Current Bead / Attempts — read-only. Clicking a row drills into live-streaming logs and recent events, but offers no actions there either.&#xA;&#xA;This raised a fair question: **what is this screen for, and how is it different from Sessions?** The answer isn&#39;t in the UI.&#xA;&#xA;## Ground truth (from the CLI)&#xA;&#xA;- **Worker** = a persistent execute-loop process draining the bead queue (`ddx work` / `ddx agent execute-loop`). Has a lifecycle: running / idle / stopped / error. Has a *current bead* and cumulative attempts/successes/failures. Can be stopped via `ddx agent workers stop`. Long-lived.&#xA;- **Session** = a historical record of one agent invocation (harness, model, bead, tokens, cost, outcome, start/end). Immutable after completion. One worker produces many sessions over its lifetime.&#xA;&#xA;So: Workers are *processes you operate*. Sessions are *events you review*. The current UI exposes both as flat tables and lets the reader guess which is which.&#xA;&#xA;## Gaps&#xA;&#xA;1. **No disambiguation.** No page-level copy, no link between the two, no worker→sessions navigation (&#34;show sessions produced by this worker&#34;), no session→worker backlink.&#xA;2. **No lifecycle controls on Workers.** The CLI has `ddx agent workers stop` and `ddx work` (start). The UI has neither button. No stop, start, restart, or &#34;cancel current bead&#34; affordance. The State column is a spectator sport.&#xA;3. **No worker start UX.** There is no way to start a queue drain from the UI — users must drop to a terminal.&#xA;4. **Unclear empty state.** &#34;No workers found.&#34; does not say whether that is normal (nothing running) or broken (server can&#39;t see workers). No hint about how to start one.&#xA;5. **No worker details beyond logs.** Worker detail page shows live phase + log tail + tool calls but not: worker config (project path, profile/harness, started-at, last-heartbeat), resource usage, or the list of sessions this worker has produced.&#xA;&#xA;## Proposed direction&#xA;&#xA;Split into two deliverables — do them together if we&#39;re touching this screen anyway, but they are independently landable:&#xA;&#xA;### A. Information architecture (copy + navigation)&#xA;&#xA;- Add a short header description to each page: &#34;Workers drain the bead queue. Sessions are the history of what they ran.&#34;&#xA;- Mutual links: Workers page links to &#34;Recent sessions →&#34;, Sessions page links to &#34;Workers →&#34;.&#xA;- On worker detail: a &#34;Sessions&#34; tab or section listing sessions where `session.workerId == this.id` (requires session records to carry workerID; check whether they do).&#xA;- On session detail (expandable row already exists): if the session has a workerID, surface it as a link back to the worker.&#xA;&#xA;### B. Lifecycle controls&#xA;&#xA;- **Stop** button per running worker row → calls a GraphQL mutation that invokes the equivalent of `ddx agent workers stop &lt;id&gt;`. Confirms before acting.&#xA;- **Start worker** button in the page header → opens a small form (project, profile/harness, effort, optional bead filter) and calls a `startWorker` mutation that shells out to `ddx work` in the background. The form should default to sane values and be submittable in one click for the common case.&#xA;- **Cancel current bead** action on the worker detail page — signals the worker to abandon the in-flight bead without killing the loop. (Only if CLI primitive exists; otherwise descope and note in the bead.)&#xA;- All mutations must be idempotent and log an audit record viewable in the UI (who did it, when). Reuse existing session-event plumbing if possible.&#xA;&#xA;### Out of scope&#xA;&#xA;- Rewriting execute-loop semantics.&#xA;- Cross-project worker orchestration.&#xA;- Auth / permissions model for who can stop workers (assume single-operator for now; file a follow-up if multi-operator becomes real).
    </description>
    <acceptance>
**User story:** As an operator looking at the DDx web UI, I want to understand at a glance what Workers are vs Sessions, and I want to stop a misbehaving worker or start a new queue drain without dropping to a terminal, so the UI is a real operator surface instead of a read-only status page.&#xA;&#xA;**Acceptance criteria:**&#xA;&#xA;1. **Copy + IA.** Workers page header includes a one-sentence description distinguishing it from Sessions. Sessions page likewise. Each links to the other. Playwright asserts both descriptions are present and the cross-links navigate.&#xA;&#xA;2. **Worker → Sessions navigation.** On worker detail, a &#34;Sessions&#34; section lists sessions produced by that worker (join on `session.workerId`). Empty state reads &#34;No sessions recorded yet.&#34; If session records do not yet carry a worker ID, that gap is filled as part of this bead and backfilled for existing sessions in the current run (no historical backfill required).&#xA;&#xA;3. **Session → Worker backlink.** Expanded session row shows worker ID as a clickable link when present.&#xA;&#xA;4. **Stop worker.** Each row with `state == &#39;running&#39;` shows a Stop button. Clicking prompts for confirm, then fires a GraphQL mutation that stops the worker. UI updates within 2s via the existing subscription/poll path. Integration test asserts: button visibility is gated on state, confirm dialog is required, and mutation invokes the same code path as `ddx agent workers stop`.&#xA;&#xA;5. **Start worker.** A &#34;Start worker&#34; button in the Workers page header opens a form (harness/profile, effort, optional label filter). Submitting fires a `startWorker` mutation. On success the new worker appears in the list within 2s. Form validation prevents empty submissions. The code path reuses `ddx work` / execute-loop internals — not a separate implementation.&#xA;&#xA;6. **Audit trail.** Every lifecycle action (start, stop, cancel) is recorded as a timestamped event visible on the worker detail page, including the actor identifier the server has for the requester.&#xA;&#xA;7. **No regressions.** Existing live log streaming, recent events, and phase overrides continue to work. Playwright smoke of the worker detail page passes.&#xA;&#xA;8. **Playwright e2e.**&#xA;   - Navigates to Workers page, asserts header description.&#xA;   - Clicks &#34;Start worker&#34;, fills form, submits, asserts new row appears.&#xA;   - Clicks Stop on that row, confirms, asserts row state transitions to `stopped`.&#xA;   - Clicks into the stopped worker, asserts the Sessions section renders (or the empty state).&#xA;   - Asserts the &#34;Sessions&#34; link in the worker page header navigates to the Sessions page.&#xA;&#xA;9. **Design review sign-off.** Before implementation begins, a short design note (one page or less) in `docs/helix/02-design/` captures the Workers-vs-Sessions mental model, the chosen controls, and any CLI primitives that need to exist. Implementation PR links to that note.
