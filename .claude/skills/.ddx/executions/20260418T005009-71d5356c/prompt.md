No PID is surfaced, so external tooling&#xA;cannot cleanly target them either.&#xA;&#xA;## Proposed shape&#xA;&#xA;```&#xA;ddx agent workers stop &lt;worker-id&gt;        # graceful stop (SIGTERM, then SIGKILL after grace)&#xA;ddx agent workers stop --all-over 1h      # stop every running worker older than 1h&#xA;ddx agent workers stop --state running    # stop all running workers&#xA;ddx agent workers stop --bead &lt;bead-id&gt;   # stop the worker assigned to a bead&#xA;```&#xA;&#xA;Graceful path:&#xA;1. Send SIGTERM to worker process group&#xA;2. Wait configurable grace period (default 30s)&#xA;3. Send SIGKILL if still alive&#xA;4. Mark worker state as &#39;stopped&#39; (distinct from &#39;failed&#39; and &#39;exited&#39;)&#xA;5. Run worker cleanup hook (worktree teardown, bead state revert if still claimed)&#xA;&#xA;Also surface PID in `ddx agent workers --json` so external tooling can target&#xA;processes directly when the CLI path is unavailable.&#xA;&#xA;## Related&#xA;&#xA;Observed on agent project 2026-04-17: nine workers running 33h–142h with no&#xA;bead assigned; root cause unclear. Even if the root cause gets fixed, operators&#xA;need a supported way to reap them.&#xA;&#xA;## Files likely touched&#xA;&#xA;- cmd/agent/workers.go — add stop subcommand and pid JSON field&#xA;- internal/agent/worker/registry.go — surface pid; add Stop() with graceful-then-force semantics&#xA;- internal/agent/worker/process.go — process group signaling, grace timer&#xA;- worker state machine — add &#39;stopping&#39; and &#39;stopped&#39; states
