    <description>
Same situational-awareness gap as the workers view: when N projects are in flight, the operator currently can only see one project&#39;s bead queue at a time via /nodes/:nodeId/projects/:projectId/beads. Add a node-level combined beads view that merges all projects&#39; queues into one filterable table.\n\nBacking data: needs a new endpoint OR client-side merge. The cleanest design is server-side aggregation since clients shouldn&#39;t fan out N requests:\n- New endpoint GET /api/beads (node-wide, no project scope)\n- Returns: array of beads with an extra &#39;project_id&#39; field on each entry\n- Supports the same query params as /api/projects/:project/beads (?status=, ?ready=, ?label=, ?limit=)\n- Walks the project registry from FEAT-020 and concatenates each project&#39;s bead store\n\nUI requirements:\n- Route: /nodes/:nodeId/beads (combined; the existing /nodes/:nodeId/projects/:projectId/beads stays)\n- Adds &#39;Project&#39; column showing project short name\n- Filter chips for status (open/in_progress/closed) and project (multi-select)\n- Filter state encoded in URL query params so views are bookmarkable\n- Auto-refresh every ~10s (slower than workers since beads change less frequently)\n\nDelete-aware: when a project is removed from the registry its beads disappear from the combined view on the next refresh.
    </description>
