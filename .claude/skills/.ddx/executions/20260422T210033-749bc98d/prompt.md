    <title>Personas UI is read-only with no explanation: decide library-vs-project policy and add lifecycle UX</title>
    <description>
## Observed&#xA;&#xA;`/nodes/.../projects/.../personas` is a well-crafted browser + bind UX, but it is read-only: no add, no edit, no delete, no explanation of what the reader is looking at or why they cannot change it. A user looking at it can reasonably ask &#34;what is this for?&#34; and &#34;why can&#39;t I change anything?&#34; and the UI answers neither.&#xA;&#xA;## Ground truth&#xA;&#xA;- **What a persona is.** A markdown file with frontmatter under `library/personas/*.md` defining AI personality, expertise, principles, communication style. Shipped defaults: architect, code-reviewer, implementer, specification-enforcer, test-engineer.&#xA;- **How it&#39;s used.** Projects declare role→persona bindings in `.ddx/config.yaml` via `persona_bindings`. When an agent performs role X, the bound persona&#39;s body is injected into its system prompt.&#xA;- **What the tool exposes today.**&#xA;  - CLI: `ddx persona --list`, `--show &lt;name&gt;`, `--bind &lt;persona&gt; --role &lt;role&gt;`. No create/edit/delete.&#xA;  - GraphQL: `personaBind` mutation only (`cli/internal/server/graphql/resolver_feat008.go:150`). No `personaCreate/Update/Delete` in schema.&#xA;  - UI: list/show/bind parity with CLI.&#xA;- **Why it evolved this way.** Personas have been treated as *library content* — authored carefully, version-controlled upstream, pulled into projects via `ddx update` and contributed back via `ddx contribute`. The tool deliberately did not become a persona editor.&#xA;&#xA;## The gap&#xA;&#xA;The library-only model doesn&#39;t hold up for real use:&#xA;&#xA;1. **Project-local personas are a real need.** Teams want &#34;our code-reviewer, with our opinions about our codebase.&#34; Today they either (a) fork the persona file and push upstream (wrong — it&#39;s not generally useful), (b) never customize (bad outcomes), or (c) drop a file into some ad-hoc local path and hope it works.&#xA;2. **The UI implies capability it doesn&#39;t have.** A list + detail view with no hint about source policy reads as &#34;unfinished feature&#34; rather than &#34;intentional read-only&#34;.&#xA;3. **Binding alone isn&#39;t enough.** Even if editing stays out of scope, users need to understand *which personas are available*, *where they come from* (library vs project-local), and *how to change that*.&#xA;&#xA;## Proposed direction&#xA;&#xA;Two decisions (pick before implementation):&#xA;&#xA;### D1 — Policy: who owns persona content?&#xA;&#xA;Recommend: **both, with a clear boundary.**&#xA;- Library personas: `library/personas/*.md` — read-only in UI, contribute upstream via `ddx contribute`.&#xA;- Project-local personas: new directory `.ddx/personas/*.md` — fully editable in UI, live with the project, not in the shared library. Override semantics: a project-local persona with the same name as a library persona takes precedence for that project&#39;s bindings.&#xA;&#xA;### D2 — Scope of the UI&#xA;&#xA;Minimum viable, given D1:&#xA;- **Explainer copy** on the personas page header: &#34;Personas are AI personality templates. Library personas are shared; project personas live with this project. Bind a persona to a role in `.ddx/config.yaml`.&#34;&#xA;- **Source badge** on each persona row: `library` or `project`.&#xA;- **Project-local editor.** For project personas:&#xA;  - `New persona` button → form (name, roles/tags, body markdown). Creates `.ddx/personas/&lt;name&gt;.md` with frontmatter.&#xA;  - `Edit` on project rows.&#xA;  - `Delete` on project rows with confirm.&#xA;- **Library rows stay read-only.** Action on a library row: `Fork to project` — copies the library file to `.ddx/personas/&lt;name&gt;.md` (with a default rename suggestion on collision) and opens the editor. No in-place edit, no delete.&#xA;- **Bind affordance remains.** Binding UI continues to work for both sources.&#xA;&#xA;## What this bead does NOT decide&#xA;&#xA;- Multi-project persona sharing (e.g., org-level personas between library and project). File a follow-up if it becomes pressing.&#xA;- Rich editor affordances beyond a plain markdown textarea with preview. Frontmatter can be a YAML-ish textarea for v1.&#xA;- Syncing project personas upstream — that remains `ddx contribute` and is out of scope.&#xA;- Any change to how personas are injected into agent prompts (the read path on the server).&#xA;&#xA;## CLI parity&#xA;&#xA;Whatever the UI gains, the CLI gains too. Add:&#xA;- `ddx persona new &lt;name&gt;` — creates `.ddx/personas/&lt;name&gt;.md` scaffold.&#xA;- `ddx persona edit &lt;name&gt;` — opens `$EDITOR` on the project-local persona (errors on library).&#xA;- `ddx persona fork &lt;library-name&gt; [--as &lt;new-name&gt;]` — copies library → project.&#xA;- `ddx persona delete &lt;name&gt;` — project-local only.&#xA;&#xA;UI and CLI call the same underlying `cli/internal/persona` operations; no duplicate logic.
    </description>
    <acceptance>
**User story:** As a developer looking at the personas UI, I want to understand at a glance what personas are, where they come from (library vs project), and I want to fork or create a project-local persona without leaving the UI. Library content stays read-only so I cannot accidentally mutate shared content; project content is fully editable.&#xA;&#xA;**Acceptance criteria:**&#xA;&#xA;1. **Design note + policy decision recorded.** Before implementation, a one-page note in `docs/helix/02-design/` captures D1 (library vs project policy, override rules, file locations) and D2 (scoped UI additions). Implementation PR links to it.&#xA;&#xA;2. **Project-local persona directory.** `.ddx/personas/*.md` is a recognized location. `ddx persona --list` shows both library and project personas with a `source` tag. Override rule: a project persona with the same name as a library persona wins for that project&#39;s bindings; unit test covers this precedence.&#xA;&#xA;3. **GraphQL additions.** New mutations: `personaCreate(name, body, projectId) → Persona`, `personaUpdate(name, body, projectId) → Persona`, `personaDelete(name, projectId) → {ok}`. All gated on `source == &#34;project&#34;`; attempting any of them on a library persona returns a typed error. Each mutation persists to `.ddx/personas/&lt;name&gt;.md` under the given project.&#xA;&#xA;4. **CLI parity.** `ddx persona new`, `edit`, `fork`, `delete` exist and share the same underlying operations as the GraphQL mutations (no duplicated logic). Integration test asserts that a `ddx persona new` followed by a GraphQL `personas` query returns the new persona.&#xA;&#xA;5. **UI — explainer copy.** The personas page shows a one-sentence description of what personas are and a second sentence distinguishing library vs project sources. Playwright asserts both sentences are present.&#xA;&#xA;6. **UI — source badge.** Each persona row shows a `library` or `project` badge. Playwright seeds one of each and asserts correct badges.&#xA;&#xA;7. **UI — New / Edit / Delete for project personas.** A &#34;New persona&#34; button opens a form with name + body fields. Project-source rows have Edit and Delete affordances. Library-source rows have neither. Playwright creates a persona, edits it, verifies the edit persisted, then deletes it.&#xA;&#xA;8. **UI — Fork library to project.** Library rows have a &#34;Fork to project&#34; action. Clicking it prompts for a name (default: library name with `-local` suffix on collision), creates `.ddx/personas/&lt;name&gt;.md`, and navigates to the editor. Playwright asserts the forked file exists and opens in the editor.&#xA;&#xA;9. **No regressions.** Existing list/show/bind flows still work. Playwright smoke of the pre-existing binding UX passes.&#xA;&#xA;10. **Empty project-local state.** When a project has zero project-local personas, the UI shows a subtle hint: &#34;No project personas yet. Fork a library persona or create a new one.&#34; Playwright asserts.
    </acceptance>
