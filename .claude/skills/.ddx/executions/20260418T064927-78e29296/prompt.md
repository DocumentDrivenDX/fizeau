## Problem (filed per codex + opus fresh-eyes review 2026-04-17)&#xA;&#xA;Both reviews flagged the same missing concept: routing currently scores candidates on cost class, quota, and historical success rate — but does NOT gate on whether the resolved route can actually execute the requested work. Concrete failure modes this lets through:&#xA;&#xA;1. **Context-window overflow.** Bead prompt is 80k tokens. Fuzzy match picks a 32k-context model on the pinned provider. Routing &#39;succeeds&#39;. Dispatch fails mid-stream with a provider 400 or truncated output.&#xA;&#xA;2. **Missing tool-calling support.** Some omlx flavors or older local servers don&#39;t implement /v1/chat/completions with tools. Routing picks them via fuzzy match. Dispatch fails with &#39;tools not supported&#39; or silently returns a no-tool response that the execute-bead loop can&#39;t use.&#xA;&#xA;3. **Missing structured-output support.** Some provider/model combos don&#39;t implement JSON-mode / tool-use-required. If the bead prompt relies on it, routing-level blindness means the failure only surfaces at dispatch.&#xA;&#xA;4. **Effort / permissions.** --effort high or --permissions unrestricted may not be honored by every harness+provider combination. Routing should reject or downgrade rather than dispatch and fail.&#xA;&#xA;Codex: &#39;routing currently scores mostly on cost/quota/history, not whether a provider/model can satisfy tool calling, effort controls, permissions, or context-window needs.&#39;&#xA;Opus: &#39;If the bead prompt is 80k tokens and the fuzzy-matched model is 32k, routing succeeds and dispatch fails mid-stream. Omlx flavor may not support tools at all.&#39;&#xA;&#xA;## Out of scope for this bead&#xA;&#xA;This bead tracks the CONCEPT as a sibling of the main autorouting epic, but a full capability-constraint scoring system is a larger change than the current Phase-1 scope. Per opus: &#39;Scope this out of the epic explicitly (don&#39;t expand), but add a sibling bead so it&#39;s tracked.&#39;&#xA;&#xA;Split out into sub-scope bites is TBD:&#xA;&#xA;1. **Context-window gating** — smallest useful slice. RouteRequest gains an optional EstimatedPromptTokens field; evaluateCandidate rejects (or penalizes) candidates whose concrete_model&#39;s known context window is below that. Uses v0.3.11&#39;s LookupModelLimits + catalog ContextWindow fallback.&#xA;&#xA;2. **Tool-calling support gating** — needs per-provider / per-flavor capability metadata. Likely populated from discovery (/v1/models/status) + hand-curated capability table for harnesses.&#xA;&#xA;3. **Effort / permissions gating** — mostly already in place for harnesses (Harness.EffortFlag checked at routing.go:rejectIfNotViable) but provider-specific gating (e.g., &#39;openrouter doesn&#39;t honor effort on non-reasoning models&#39;) is not.&#xA;&#xA;4. **Structured-output support** — probably lowest priority; most reasoning-tool-capable models also do structured output.&#xA;&#xA;This bead&#39;s AC is to file the sub-beads (smaller, focused) and close THIS bead once the sub-beads exist. The epic ddx-2d974641 does NOT depend on those sub-beads for its exit gate — context-window etc. are a Phase 2 concern. The epic is about correct resolution; capability gating is about preventing resolved-but-unrunnable routes.&#xA;&#xA;## Files touched (by sub-beads, not this one)&#xA;&#xA;- cli/internal/agent/types.go — CandidatePlan capability fields, RouteRequest capability hints&#xA;- cli/internal/agent/routing.go — capability checks in rejectIfNotViable + scoring&#xA;- cli/internal/agent/catalog.go — ContextWindow, supports_tools, supports_structured_output per ModelEntry/Surface
