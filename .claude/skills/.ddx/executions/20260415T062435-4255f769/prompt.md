The ddx-bead skill guides bead creation but underspecifies what makes a bead *execution-ready* for automated execute-bead runs. Observed during axon ADR-018 execute-loop driving: beads that technically met the current skill&#39;s requirements (title, AC, labels, description pointing at spec-id) still produced bad execute-bead outcomes because the description lacked the context the agent needed to ship correct code in one pass.&#xA;&#xA;## Gaps in the current skill&#xA;&#xA;1. **Description quality is underspecified.** The skill gives AC quality rules but the description field just says &#39;per SD-004 Section 2&#39; is OK. In practice, agents need:&#xA;   - Exact file paths to touch (e.g., &#39;crates/axon-server/src/auth_pipeline.rs&#39;)&#xA;   - Cross-bead context when dependencies are listed (e.g., &#39;A1 landed the credential_revocations table in crates/axon-storage/src/auth_schema.rs — query it via SELECT 1&#39;)&#xA;   - Inline spec excerpts, NOT just doc-id references. If the spec is ADR-018 §4&#39;s failure table, paste the table into the description. Don&#39;t say &#39;see ADR-018 §4&#39;.&#xA;   - Files NOT to touch (scope negation)&#xA;&#xA;2. **spec-id and the doc graph are unreliable for execution context.** The current skill implies spec-id linkage is sufficient. It is not — spec-id drives docgraph.BuildGraphWithConfig() which only resolves HELIX-shaped doc references. Generic beads on non-HELIX projects have no equivalent. AND even when resolution works, the agent only gets a file path, not the relevant section. The bead description is the only place that carries content the agent can rely on.&#xA;&#xA;3. **No guidance on cross-bead context.** When bead B depends on bead A, B&#39;s description should describe what A landed (file paths, type names, key APIs) — not assume the agent will reopen A&#39;s history. Automated execute-bead does not have that context; each bead is read cold.&#xA;&#xA;4. **No guidance on verification commands in AC.** AC items like &#39;unit tests green&#39; are too vague. Should be &#39;cargo test -p axon-server green&#39; or &#39;bun test passes in sdk/typescript&#39;. Named commands.&#xA;&#xA;5. **No guidance on bead size.** A bead should be one unit of testable work, sized so a single agent pass can complete it. Too small is overhead; too large defeats review. Heuristic: if the AC lists more than ~12 checkboxes, consider splitting.&#xA;&#xA;6. **No guidance on incorporating review findings on reopen.** When a bead is reopened after a REQUEST_CHANGES review, the author should either (a) fold the review findings into the description for the next attempt to see, or (b) rely on the tool to thread review events into the prompt (a separate ddx bead — see ddx-a317208c). The skill is silent on this.&#xA;&#xA;7. **No template for description.** A recommended description template (Scope / Files to touch / Cross-bead context / Verification / Quality bar) would make good beads cheaper to author.&#xA;&#xA;## Scope&#xA;&#xA;Update skills/ddx-bead/SKILL.md to add:&#xA;&#xA;- A &#39;What makes a bead execution-ready&#39; section that codifies the points above&#xA;- A &#39;Description template&#39; with recommended sections: Scope, Files to touch, Files NOT to touch, Cross-bead context, Verification commands, Spec excerpts inline&#xA;- A &#39;Bead sizing&#39; section with the ~12-AC heuristic&#xA;- A &#39;Review-driven reopen&#39; section that tells authors to update the description in place when a bead is reopened after a review rejection&#xA;- A &#39;Common anti-patterns&#39; section enumerating the mistakes observed during the axon execute-loop:&#xA;  - Listing doc IDs without inlining the relevant text&#xA;  - Not naming file paths in the description&#xA;  - AC items that are not verifiable by a named command&#xA;  - Depending on unmerged work without explaining what it landed&#xA;&#xA;Keep the existing required-fields and label-conventions sections — they are correct.&#xA;&#xA;## Context&#xA;&#xA;Observed during axon ADR-018 execute-loop:&#xA;- Bead A1 was well-sized but initially had scope creep in its description (audit_retention_policies, which belonged to F1). Agent added the scope-creep table faithfully.&#xA;- Bead A2&#39;s initial description said &#39;UserIdentity { provider, subject, user_id }&#39; — a typo carried over from an earlier ADR revision. Agent implemented &#39;subject&#39; faithfully even though ADR-018 §3 said &#39;external_id&#39;. The description was the contract the agent followed.&#xA;- Bead B1 is being re-authored right now with a richer description (inline ADR-018 §4 failure table, exact file paths, cross-bead context about A1 schema and A2 types) — this is the experiment to validate the authoring changes proposed in this bead.&#xA;&#xA;## Acceptance&#xA;&#xA;- [ ] skills/ddx-bead/SKILL.md gains &#39;What makes a bead execution-ready&#39; section&#xA;- [ ] Description template added with the 6 recommended sections&#xA;- [ ] Bead sizing heuristic documented&#xA;- [ ] Review-driven reopen guidance added&#xA;- [ ] Common anti-patterns enumerated&#xA;- [ ] Skill still lints against any existing skill-format CI check
