    <title>execute-bead: autonomous worker watchdog with process-level reaper</title>
    <description>
## Residual risk after RC1–RC4 from ddx-0a651925&#xA;&#xA;Even with context propagation (RC1), wall-clock bounds (RC2), tightened breakers (RC3), and provider deadlines (RC4), a goroutine-level deadlock or panic-swallowed handler could still leave a worker alive past any reasonable bound. Today, `server.WorkerManager.Stop` (workers.go:648) only calls `handle.cancel()` — it has no escalation to OS-level signal and no autonomous trigger. ddx-b808df39 adds the *operator* tool; this bead adds the *autonomous* watchdog.&#xA;&#xA;## Scope&#xA;&#xA;1. WorkerManager runs a supervisor goroutine that, every minute, inspects each `workerHandle`.&#xA;2. If `time.Since(record.StartedAt) &gt; WatchdogDeadline` (default e.g. 6h) AND `record.CurrentAttempt` has not transitioned in `StallDeadline` (default e.g. 1h), escalate:&#xA;   - Send SIGTERM to the worker&#39;s process group (requires surfacing PID, aligned with ddx-b808df39).&#xA;   - After 30s grace, send SIGKILL.&#xA;   - Mark record.State = &#34;reaped&#34;; update claimed bead state so the claim is not leaked.&#xA;3. Emit a `bead.reaped` event on the tracker with the full timeline so post-mortems are easy.&#xA;&#xA;## Non-goals&#xA;&#xA;- Per-bead runtime tuning; defaults are enough.&#xA;- Replacing ddx-b808df39 — this is defense-in-depth, not duplication. ddx-b808df39 is operator-initiated; this bead is system-initiated.&#xA;&#xA;## Dependencies&#xA;&#xA;- ddx-b808df39 surfaces PID in `ddx agent workers --json` output; this bead reuses that plumbing.
