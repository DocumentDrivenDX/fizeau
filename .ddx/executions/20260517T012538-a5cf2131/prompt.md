PROBLEM&#xA;The credit probe in [[openrouter-credit-balance-probe-and-cache]] can fail in two distinct ways that operators need to distinguish:&#xA;- 401 from `/api/v1/credits` → the key is set but rejected (e.g. revoked, typo&#39;d). This is different from &#34;never set a key&#34; because the fix is different (rotate vs configure).&#xA;- Transport failure (DNS, TCP, TLS, 5xx) → openrouter.ai is unreachable. This is transient and we must not hard-exclude openrouter forever, or one bad minute permanently demotes the provider.&#xA;&#xA;Without this distinction operators cannot triage; without fail-open recovery a transient blip causes flapping or sustained mis-routing.&#xA;&#xA;ROOT CAUSE&#xA;The probe path introduced by [[openrouter-credit-balance-probe-and-cache]] needs explicit failure-classification branches and a freshness-window policy. The error handling for the openrouter credits roundtripper response must inspect status code and error type, then write a typed `filter_reason` into the cache rather than a generic error string. Exact integration point: the cache-write call in the credit-probe routine.&#xA;&#xA;PROPOSED FIX&#xA;1. **401 from /credits** → cache `filter_reason: &#34;credential_invalid&#34;` (distinct from `&#34;credential_missing&#34;` emitted by [[openrouter-credential-presence-gate]]). Candidate `eligible: false`. Operators reading routing_decision can tell &#34;forgot to set&#34; from &#34;key rejected&#34;.&#xA;2. **Transport error / non-401 5xx** → cache `filter_reason: &#34;provider_unreachable&#34;`. Candidate `eligible: false` for the current freshness window ONLY — do not hard-exclude. Next scheduled probe pass attempts a fresh call; on success the candidate returns to `eligible: true` automatically.&#xA;3. **Fail-open semantics**: a `provider_unreachable` cache entry must NOT be sticky beyond the probe interval. After the next probe attempt the cache reflects the new state (success or another failure-class). No manual operator intervention required to recover.&#xA;4. **Evidence in routing_decision**: each filter_reason carries the originating HTTP status code (or transport error class) in the candidate evidence body so the operator does not have to grep logs.&#xA;&#xA;NON-SCOPE&#xA;- Credential presence/format checks (handled by [[openrouter-credential-presence-gate]]).&#xA;- The credit balance probe and its caching layer itself (handled by [[openrouter-credit-balance-probe-and-cache]]).&#xA;- Retry-with-backoff against openrouter.ai on transport failure during the probe — the probe is best-effort; the next scheduled pass is the retry.&#xA;- Multi-account credential rotation / failover (parent non-scope, preserved).&#xA;- Per-model credit-cost forecasting (parent non-scope, preserved).&#xA;- DDx-side changes per CONTRACT-003 (parent non-scope, preserved).&#xA;&#xA;CROSS-REF&#xA;Sibling beads: [[openrouter-credential-presence-gate]], [[openrouter-credit-balance-probe-and-cache]].
