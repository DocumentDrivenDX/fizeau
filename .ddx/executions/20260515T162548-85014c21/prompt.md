    <title>routing: persist provider-health signals across ddx process boundaries</title>
    <description>
PROBLEM&#xA;internal/routehealth.Store tracks per-route attempt feedback (failures, cooldowns, sticky leases) inside a single Fizeau process. ddx work and ddx try are short-lived processes that each instantiate their own Store on startup, so health learned during one ddx invocation is invisible to the next. Observed today: bragi:1234 was unreachable. Worker 1&#39;s ddx try learned that, recorded a route-failure, and excluded bragi for its remaining attempts. When worker 1 finished and worker 2&#39;s next ddx try started, the new Fizeau process had a fresh empty routehealth.Store and immediately re-routed to bragi — same dial timeout, same failure. The exclusion list visible via ddx-side work-failed-routes (which IS persisted in .ddx/beads.jsonl per bead) helps within a bead&#39;s attempts but not for cross-bead provider health: every new bead starts assuming bragi is healthy, eats 5-30s of dial timeout, then learns again.&#xA;&#xA;ROOT CAUSE&#xA;service.go:974-976 (NewService) constructs routehealth.NewStore() / NewLeaseStore() / NewUtilizationStore() with no persistence backing. The package internal/routehealth has no on-disk format; everything is in-process maps. There is no documented contract for whether health signals should persist, expire, or be considered process-local by design.&#xA;&#xA;PROPOSED FIX&#xA;- Add an optional ServiceOptions field PersistRouteHealth string (path or directory) that, when set, causes routehealth.Store to read/write a small JSON or jsonl health snapshot on shutdown and at periodic intervals (default 60s).&#xA;- Snapshot scope: per-route failure counts, cooldown-until timestamps, success counters. Treat the persistence as best-effort; missing/corrupt file ⇒ start with empty state and log a warning.&#xA;- Health snapshots expire on read: any entry whose recorded timestamp is older than HealthSignalTTL (default 10 min) is dropped. So a brief outage learned by one process expires before the next process even if the file is not rewritten.&#xA;- DDx side will set PersistRouteHealth to &#34;.ddx/routehealth.json&#34; so all ddx work / ddx try / ddx run invocations within a project share the signal. Filed in companion bead.&#xA;&#xA;NON-SCOPE&#xA;- Do not change the in-process routehealth API surface; persistence is additive.&#xA;- Do not introduce a network-backed health store (out of scope; persistence is per-machine).&#xA;- Do not change cooldown semantics; this is only about cross-process visibility of what was already learned.&#xA;&#xA;DEPS&#xA;None.
    </description>
