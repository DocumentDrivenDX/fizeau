    <title>bench: kill fiz harness descendants on Harbor timeout</title>
    <description>
PROBLEM&#xA;A timed-out or cancelled TerminalBench Harbor run can leave Fizeau harness subprocesses alive after Harbor reports the cell as failed. In the local incident on 2026-05-16, the OrbStack VM had 610 gemini-matching processes and the process/swap aggregate showed `gemini` holding about 69,659 MiB total (22,929 MiB RSS + 46,730 MiB swap). System memory was 60/62 GiB used, swap was effectively full, and `/sys/fs/cgroup/memory.events` showed `oom_kill 132`. Cleaning up the Gemini/Node processes immediately raised available RAM from about 2 GiB to about 28-36 GiB.&#xA;&#xA;ROOT CAUSE WITH FILE:LINE&#xA;- `scripts/benchmark/harbor_agent.py:347-356` builds a bash `cmd=(/installed-agent/fiz ...)` array and runs it as `&#34;${cmd[@]}&#34; 2&gt;&amp;1 | stdbuf -oL tee /logs/agent/fiz.txt`, then reads `PIPESTATUS[0]`. There is no `trap` for TERM/INT/EXIT and no child-process-tree cleanup. If Harbor cancels the Docker exec shell on AgentTimeout, the shell can die while `fiz` or its wrapped subprocess harness (`gemini`, `claude`, `codex`, etc.) remains orphaned.&#xA;- `agentcli/run.go:339-341` creates the execution context with `signal.NotifyContext(context.Background(), os.Interrupt)` only. A SIGTERM from Harbor/runner cancellation is not converted into service context cancellation, so subprocess harness cleanup such as `internal/harnesses/gemini/runner.go:274-287` may not run before the `fiz` process exits.&#xA;- `cmd/bench/matrix.go:1097-1104` sends SIGTERM to the Harbor process and gives it 60s `WaitDelay`, but that only assumes Harbor and Docker Compose finalizers clean up the trial. It does not verify or enforce cleanup of the nested `fiz -&gt; harness CLI` process tree that the FizeauAgent shell wrapper starts inside the task container.&#xA;&#xA;PROPOSED FIX&#xA;Make cancellation cleanup explicit at both levels:&#xA;1. Teach the `fiz` CLI to treat SIGTERM like os.Interrupt for run cancellation on Unix, so the service context is cancelled and harness runners can kill their subprocess groups before the process exits.&#xA;2. Rewrite the FizeauAgent generated bash in `scripts/benchmark/harbor_agent.py` so it wraps the `fiz` invocation with TERM/INT/EXIT cleanup. Preserve tee logging and the `fiz_rc` status, but keep the child PID, terminate the child process tree on cancellation, wait a short grace interval, then SIGKILL leftovers. Avoid a bare foreground pipeline where killing the shell loses the child PID.&#xA;3. Add or update benchmark-runner cancellation tests so parent cancellation exercises the cleanup path and the generated command structure cannot regress to an untrapped pipeline.&#xA;&#xA;IN-SCOPE FILES&#xA;- `agentcli/run.go`&#xA;- `agentcli/cli_process_test.go`&#xA;- `scripts/benchmark/harbor_agent.py`&#xA;- `scripts/benchmark/test_harbor_agent.py`&#xA;- `cmd/bench/matrix.go` and `cmd/bench/matrix_test.go` only if runner-level process-group cancellation must change to make the cleanup complete.&#xA;&#xA;NON-SCOPE&#xA;- Rewriting the benchmark runner beyond cancellation/cleanup behavior.&#xA;- Changing TerminalBench task Docker images.&#xA;- Changing Gemini/Codex/Claude harness routing or model selection.&#xA;- Bulk deleting existing leaked processes; this bead is for preventing recurrence.&#xA;&#xA;NOTES FOR IMPLEMENTER&#xA;The existing `TestCLI_CancelSignal_WritesSessionEndEvent` covers os.Interrupt. Add a SIGTERM equivalent rather than replacing that test. The Python command-builder tests already mock `exec_as_agent`, so they are the right place to assert the generated bash contains the trap/cleanup structure without needing a live Harbor run.
