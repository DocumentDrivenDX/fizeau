    <title>Derive reasoning_tokens from reasoning_content when usage.completion_tokens_details is absent</title>
    <description>
Today the reasoning-token extractor in `internal/sdk/openaicompat/client.go`&#xA;(`extractReasoningTokens`) reads only from&#xA;`usage.completion_tokens_details.reasoning_tokens` in the response body. ds4&#xA;and some other providers do thinking but do NOT populate that field — they&#xA;return the thinking inline as `message.reasoning_content` with no separate&#xA;token-count accounting.&#xA;&#xA;Verified 2026-05-11 against ds4 (vidar:1236): a simple &#34;What is 17 times&#xA;24?&#34; prompt returned `message.reasoning_content` of 92 chars but&#xA;`usage.completion_tokens_details` was absent entirely. Result: report.json&#xA;records reasoning_tokens=0 for every ds4 cell even when thinking ran for&#xA;thousands of tokens.&#xA;&#xA;Per ADR-010 Amendment §8, the resolution chain must be:&#xA;1. If usage.completion_tokens_details.reasoning_tokens is present, use it.&#xA;2. Else if message.reasoning_content is non-empty, derive an approximation&#xA;   from char-count / 4. Annotate with reasoning_tokens_approx=true.&#xA;3. Else reasoning_tokens=0 (honest zero, not silent truncation).&#xA;&#xA;Implementation:&#xA;&#xA;1. Extend `extractReasoningTokens` in&#xA;   `internal/sdk/openaicompat/client.go` to take both the raw usage JSON&#xA;   AND the raw message JSON (or refactor to read message.reasoning_content&#xA;   from the parsed response if it&#39;s already structured).&#xA;2. Apply the char-count fallback when usage path is absent AND&#xA;   message.reasoning_content is non-empty.&#xA;3. Add a new boolean field `ReasoningTokensApprox` to&#xA;   `internal/core/agent.go` `TokenUsage` struct. Default false.&#xA;4. Wire ReasoningTokensApprox through into the matrix runner&#39;s report.json&#xA;   (`cmd/bench/matrix.go` — alongside the existing ReasoningTokens&#xA;   plumbing).&#xA;5. The char-count/4 estimator is a v1 placeholder; a future bead can&#xA;   replace with a real tokenizer (tiktoken-go or model-specific) if&#xA;   needed. Document the v1 method in the field&#39;s GoDoc.&#xA;&#xA;Test cases (table-driven in&#xA;`internal/sdk/openaicompat/client_test.go`):&#xA;- usage path present, reasoning_content also present → use usage (more&#xA;  authoritative; report approx=false).&#xA;- usage path absent, reasoning_content present (92 chars) → derive 23&#xA;  tokens (92/4). approx=true.&#xA;- both absent → 0, approx=false.&#xA;- usage path = 0 (explicit zero), reasoning_content present → still use&#xA;  usage&#39;s zero (caller signaled no reasoning); approx=false. NOTE: this&#xA;  is a judgment call; document it.&#xA;- Streaming path: aggregate reasoning_content across deltas, char-count&#xA;  the aggregated string.&#xA;&#xA;In-scope files:&#xA;- internal/sdk/openaicompat/client.go&#xA;- internal/sdk/openaicompat/client_test.go (new tests)&#xA;- internal/core/agent.go (add ReasoningTokensApprox field to TokenUsage)&#xA;- cmd/bench/matrix.go (extend report.json to carry the approx flag)&#xA;&#xA;Out-of-scope:&#xA;- Real tokenizer integration (tiktoken-go etc.) — v1 uses chars/4.
