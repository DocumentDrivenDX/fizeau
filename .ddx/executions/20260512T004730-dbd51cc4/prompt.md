    <title>ADR-012: Per-source on-disk cache for discovery + runtime signals</title>
    <description>
Design ADR for the per-source on-disk cache that backs the available-&#xA;models snapshot. Defines file layout, locking semantics, single-flight&#xA;refresh algorithm, atomic-write contract, force-refresh behavior, TTLs,&#xA;crash recovery, and the mandatory test matrix for the implementation.&#xA;&#xA;This is design-only. Implementation is bead M1.&#xA;&#xA;## ADR scope (sections to write)&#xA;&#xA;### Context&#xA;- The `fiz models` snapshot needs live data from multiple sources&#xA;  (provider /v1/models, harness PTY, /props introspection, /usage&#xA;  endpoints, rate-limit response headers).&#xA;- Discovery sources are slow (PTY ~30s, OR /api/v1/models ~316ms).&#xA;- Runtime signals (quota, status, latency) change frequently.&#xA;- UI must never hang (100ms budget).&#xA;- Multi-process: `ddx work`, `ddx try`, `fiz models`, routing all run&#xA;  concurrently and must share cache state.&#xA;- fizeau already has a battle-tested file-locking idiom in&#xA;  `cmd/bench/matrix.go:acquireMatrixLock` (PID + atomic-create-or-fail&#xA;  with crash recovery via PID-alive check). Build on it.&#xA;&#xA;### Decision&#xA;&#xA;#### File layout&#xA;```&#xA;~/.cache/fizeau/   (os.UserCacheDir())&#xA;├── discovery/                          # slow, stable&#xA;│   ├── openrouter.json                 # 24h TTL&#xA;│   ├── claude-subscription.json        # 24h TTL&#xA;│   ├── codex-subscription.json         # 24h TTL&#xA;│   ├── vidar-ds4.json                  # 1h TTL (local)&#xA;│   └── sindri-club-3090-llamacpp.json  # 1h TTL (local)&#xA;└── runtime/                            # hot, volatile&#xA;    ├── openrouter.json                 # 5min TTL&#xA;    ├── claude-subscription.json        # 5min TTL&#xA;    └── ...&#xA;```&#xA;Each data file is JSON. Keys map to fizeau&#39;s canonical model identity:&#xA;`&lt;provider&gt;/&lt;model_id&gt;`.&#xA;&#xA;#### Lock + marker pattern (two-tier)&#xA;- `&lt;source&gt;.lock`: brief mutation lock. Same idiom as&#xA;  `acquireMatrixLock` — `O_CREATE|O_EXCL`, JSON `{PID, StartedAt}`,&#xA;  crash recovery via `processAlive(PID)`. Held during state&#xA;  transitions only (claim refresh, finish refresh).&#xA;- `&lt;source&gt;.refreshing`: long-lived refresh marker. JSON&#xA;  `{PID, StartedAt, Deadline}`. Held during refresh IO. Other&#xA;  processes see this and wait OR return stale.&#xA;&#xA;#### Algorithms (full pseudocode in ADR)&#xA;1. `claim_refresh(source) -&gt; ClaimedByMe | AlreadyInFlight(marker)`&#xA;2. `refresh_and_commit(source)` — IO, atomic-rename, marker cleanup&#xA;3. `wait_for_refresh(source, marker, max_wait)` — poll marker&#xA;4. `read(source) -&gt; (data, fresh_bool)` — UI never blocks&#xA;5. `force_refresh(source)` — sync; wait or claim&#xA;6. In-process single-flight via `golang.org/x/sync/singleflight`&#xA;   (composes with file-based marker for cross-process)&#xA;&#xA;#### TTL defaults (operator-configurable via env or config)&#xA;- Discovery PTY: 24h (PTY is slow; refresh rare)&#xA;- Discovery HTTP /v1/models: 24h&#xA;- Discovery HTTP /v1/models (local): 1h&#xA;- Runtime signals: 5 min&#xA;- Lock acquisition timeout: 100ms&#xA;- Refresh deadline: 60s for PTY, 10s for HTTP discovery, 5s for runtime&#xA;- Marker staleness threshold: 2× refresh deadline&#xA;&#xA;#### Crash recovery&#xA;- Stale `.refreshing` markers (past deadline OR dead PID) are&#xA;  presumed orphan; next caller overrides. No manual cleanup needed.&#xA;- Stale `.lock` files (dead PID) are removed on next attempt&#xA;  (existing `acquireMatrixLock` semantics).&#xA;&#xA;#### Force-refresh semantics&#xA;- Synchronous. Waits for in-flight refresh to complete (up to&#xA;  refresh deadline). Then reads fresh data.&#xA;- Bypasses TTL check.&#xA;&#xA;#### Cache prune&#xA;- `fiz cache prune` removes data files for sources not in current&#xA;  config. Called explicitly OR auto-pruned at startup (operator&#xA;  decision; ADR proposes manual to avoid surprises).&#xA;&#xA;### Mandatory test matrix (10 tests)&#xA;&#xA;The implementation bead (M1) MUST ship all 10 tests passing. Skipping&#xA;any = the cache is dangerous. Spawn child Go processes for multi-&#xA;process scenarios (standard `TestMain` + `os.Args[0]` -test.run&#xA;helper-process pattern).&#xA;&#xA;1. **Concurrent claim from 2 procs**: exactly one ClaimedByMe.&#xA;2. **Reader during refresh**: atomic rename → no torn read; reader&#xA;   sees old until rename, new after.&#xA;3. **Crash during refresh** (kill -9): next process detects dead PID&#xA;   in marker, claims.&#xA;4. **Refresh timeout**: marker deadline expires, next process&#xA;   claims; orig process&#39;s late commit rejected (lock check).&#xA;5. **Force-refresh waits + reads fresh**.&#xA;6. **Concurrent normal+force**: normal returns stale immediately,&#xA;   background refresh single-flighted with force.&#xA;7. **Atomic rename verified**: 100 concurrent readers + 1 writer&#xA;   producing 100 versions; every read is a complete consistent&#xA;   version (checksum).&#xA;8. **`fiz cache prune` doesn&#39;t race active sources**.&#xA;9. **Stale-while-revalidate**: stale read returns immediately,&#xA;   background refresh fires.&#xA;10. **PID reuse safety**: marker PID gets reused by unrelated&#xA;    process; deadline check is the safety net.&#xA;&#xA;### Consequences&#xA;&#xA;**Positive:**&#xA;- UI never hangs (read path is always cache-bounded &lt;
