    <title>routing: bound TCP dial timeout to ~5s, not Go default ~30s</title>
    <description>
PROBLEM&#xA;A dead local provider endpoint costs ~30 seconds of wall-clock per call before Fizeau surfaces the dial error. The fizeau-8134d6db AC required dial-class failures to bail in under 5 seconds; the shipped implementation only short-circuits the *retry loop* (one attempt instead of four), but the *dial itself* still uses Go&#39;s default TCP connect timeout (~30s on Linux). Observed today: every bead routed to bragi:1234 / bragi:8020 / grendel:1234 / grendel:1235 / grendel:8000 took ~30s to fail with &#34;dial tcp ... i/o timeout&#34;. With 14 beads in the ddx queue all routing to a single dead host, the drain spent ~7 minutes establishing that the host was dead — meeting fizeau-8134d6db&#39;s letter (single attempt) while missing its spirit (under 5s).&#xA;&#xA;ROOT CAUSE&#xA;internal/core/loop.go:643 (IsDialError short-circuit) is correctly wired. The retry-loop layer fails fast.&#xA;&#xA;What is NOT bounded: the underlying http.Transport&#39;s DialContext. internal/provider/openai/openai.go and internal/sdk/openaicompat construct openaicompat.Config without overriding the HTTP client&#39;s transport. Go&#39;s net/http default Transport has no explicit DialContext, so it uses net.Dialer{} with Timeout=0 — which on Linux waits for the kernel&#39;s full TCP SYN retransmit timeout (≈21s + RTO backoff = 30s observed). For a known-local endpoint that the operator just unplugged, 30s is pure waste.&#xA;&#xA;PROPOSED FIX&#xA;- Configure an explicit net.Dialer with Timeout=5s (or ServiceOptions-driven, defaulting to 5s) in the openaicompat client construction path.&#xA;- Wrap http.Transport.DialContext so the connect phase is bounded independent of the request streaming timeout.&#xA;- Keep the request-streaming timeout (Timeout field) separate — large requests legitimately take minutes once connected.&#xA;- Expose ConnectTimeout in openaicompat.Config so callers (DDx, benchmarks) can tune for high-latency networks if needed; default 5s.&#xA;&#xA;NON-SCOPE&#xA;- Do not change the retry-loop short-circuit (already correct).&#xA;- Do not change in-flight 5xx retry behavior.&#xA;- Do not bound the request body / streaming timeout in this bead.&#xA;&#xA;DEPS&#xA;None.
