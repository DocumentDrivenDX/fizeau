    <title>Server: dedupe and garbage-collect the project registry; tombstone unreachable projects</title>
    <description>
GET /api/projects currently returns 4186 entries on the eitri node. The real number of projects on this machine is single-digit. Root cause: the per-CLI-invocation auto-registration in serverreg.TryRegisterAsync registers the project on EVERY ddx command run with no dedupe, AND historical bugs (CWD drift in linked worktrees, the stale beads-store-write bug fixed in 26c0266) caused thousands of phantom &#39;projects&#39; to land with paths like /Users/erik/Projects/ddx/.claude/worktrees/agent-XYZ. The registry has no GC, no path canonicalization, no reachability check.\n\nFix scope:\n1. Path canonicalization on register: filepath.Abs + EvalSymlinks before storing, so /tmp and /Users/erik/Projects/.... are normalized\n2. Detect linked-worktree paths (via git rev-parse --git-common-dir) and store the PRIMARY worktree path, never the linked worktree path itself — this is the same fix we did for FindNearestDDxWorkspace in commit 26c0266 but applied to project registration\n3. Dedupe on register: if a project with the same canonical path already exists, update its last_seen instead of creating a new entry\n4. Reachability sweep: a periodic background goroutine (or a manual ddx server projects gc command) that stat()&#39;s each registered project&#39;s path and either (a) marks it as &#39;unreachable&#39; with a tombstone timestamp or (b) removes entries whose path has been gone for &gt;24h\n5. Migration: on server startup, run dedupe + path canonicalization + reachability sweep ONCE against the existing state file so the eitri 4186 entries collapse to the real count\n6. GET /api/projects gains a ?include_unreachable= query (default false) so the UI can hide tombstones unless asked
    </description>

(3) registers a project, deletes the path on disk, runs the reachability sweep (expose a Server.SweepProjects helper for tests), asserts the project is marked unreachable with a tombstone timestamp, then asserts GET /api/projects with default params hides it and ?include_unreachable=true shows it. Migration test: writes a state file with 50 duplicate entries for the same canonical path, starts the server, asserts the post-startup state file has exactly 1 entry. Acceptance smoke: running the migration on the eitri server&#39;s current state file reduces the count from 4186 to single-digit, verifiable by curl https://127.0.0.1:7743/api/projects | jq length.
