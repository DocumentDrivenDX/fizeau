    <title>Plugins page doesn&#39;t refresh after update/install: worker completes, card still shows outdated</title>
    <description>
## Observed&#xA;&#xA;On `/nodes/.../projects/.../plugins`, clicking Update on the outdated `helix` card:&#xA;&#xA;1. Fires the mutation — works.&#xA;2. Shows a link to the dispatched worker — helpful.&#xA;3. Worker completes and the plugin is actually updated on disk — verified.&#xA;4. **Card still shows `update-available`.** The UI never closes the loop.&#xA;&#xA;The user has no in-UI signal that the update succeeded. They must reload the page (or navigate away and back) to see the new state. Worse, nothing prevents them from clicking the button again during the gap — firing a second update.&#xA;&#xA;## Root cause&#xA;&#xA;`cli/internal/server/frontend/src/routes/nodes/[nodeId]/projects/[projectId]/plugins/+page.svelte:74-89` —&#xA;&#xA;```js&#xA;async function dispatchPlugin(name, action, scope = &#39;project&#39;) {&#xA;  const result = await client.request(PLUGIN_DISPATCH_MUTATION, { name, action, scope });&#xA;  workerId = result.pluginDispatch.id;&#xA;}&#xA;```&#xA;&#xA;That is the entire post-dispatch behavior. `data.plugins` is the snapshot from `+page.ts` at load time. There is no:&#xA;&#xA;- subscription to the dispatched worker&#39;s progress,&#xA;- polling of the plugin list,&#xA;- refetch on worker terminal event,&#xA;- optimistic card state while the worker runs,&#xA;- per-card in-flight tracking (only a single `workerId: string | null` at module scope, which means dispatching a second plugin overwrites the first worker link).&#xA;&#xA;The workers layout already has `subscribeWorkerProgress(workerID, cb)` (`cli/internal/server/frontend/src/routes/nodes/[nodeId]/projects/[projectId]/workers/+layout.svelte:22-30`) — the plugins page just doesn&#39;t use it.&#xA;&#xA;## Proposed direction&#xA;&#xA;Two changes, smallest first:&#xA;&#xA;### Fix A — Close the loop on the dispatched worker.&#xA;&#xA;- Per-card in-flight tracking: replace the singleton `workerId` with a `Map&lt;pluginName, workerId&gt;`. Each card shows its own worker link + in-flight state; clicking Install/Update is disabled while that plugin is in flight.&#xA;- Subscribe to `subscribeWorkerProgress` for each in-flight plugin. On a terminal phase event (`completed`, `failed`, etc. — whatever the existing worker event vocabulary uses), refetch the plugins list and clear that plugin&#39;s in-flight entry.&#xA;- If the existing worker-progress stream doesn&#39;t emit a terminal event with enough detail, fall back to polling `plugins { name status installedVersion }` every 2s while any card is in flight, stopping when none are.&#xA;&#xA;### Fix B — Optimistic card state (optional polish).&#xA;&#xA;- While a card is in-flight, show &#34;Updating…&#34; / &#34;Installing…&#34; with a spinner in place of the status badge, and disable the action button. This is orthogonal to Fix A and can ship with it.&#xA;&#xA;### Fix C — Persistent failure surfacing.&#xA;&#xA;- Currently `dispatchError` is a single string. If the worker terminates with a failure phase, surface that per-card (error badge + tooltip with detail link to the worker page) rather than relying on `dispatchError` which only covers the synchronous mutation error.&#xA;&#xA;## Out of scope&#xA;&#xA;- Designing new worker terminal events. Use whatever the workers layout already consumes.&#xA;- Cancelling an in-flight plugin update — follow-up if needed.&#xA;- Bulk update (&#34;update all outdated&#34;). Follow-up.
    </description>
    <acceptance>
**User story:** As a developer on the plugins page, I want clicking Update or Install to give me continuous feedback until the operation completes, and I want the card to reflect the new installed state automatically — without a page reload.&#xA;&#xA;**Acceptance criteria:**&#xA;&#xA;1. **Per-card in-flight tracking.** Clicking Update/Install records the workerId under the plugin&#39;s name in a `Map&lt;string, string&gt;`. The same plugin cannot be dispatched again while it is in flight (button disabled; tooltip shows the worker link). A different plugin can be dispatched concurrently and tracks its own worker.&#xA;&#xA;2. **Worker-driven refresh.** For each in-flight plugin, the page subscribes to that worker&#39;s progress via the existing `subscribeWorkerProgress` channel. On a terminal event, the page refetches the plugins list and clears that plugin&#39;s entry from the in-flight map. Within 2s of worker terminal event, the card reflects the new status (e.g., `update-available` → `installed`, new `installedVersion`).&#xA;&#xA;3. **Fallback polling.** If no terminal event arrives within 30s of dispatch, the page begins polling `plugins { name status installedVersion }` every 2s until the card&#39;s status or installedVersion changes, then stops. Integration test simulates a stream dropout and asserts the card eventually refreshes via the poll path.&#xA;&#xA;4. **In-flight visual state.** While a card is in flight, its status badge is replaced by a spinner + &#34;Updating…&#34; or &#34;Installing…&#34; text. The action button is disabled with a tooltip pointing to the worker.&#xA;&#xA;5. **Failure surfacing.** If the worker terminates in a failure phase, the card shows an error badge + tooltip &#34;Update failed — view worker&#34; linking to the worker page. The card&#39;s status/version remains whatever it was before dispatch (no false success). The global `dispatchError` stays for synchronous mutation errors only.&#xA;&#xA;6. **Concurrent dispatch.** Dispatching Update on two different cards at once shows two active worker links (a row or small stack in the header instead of the single slot), and each card tracks its own state. Playwright asserts.&#xA;&#xA;7. **No extra refetch overhead.** Refetch is scoped to the plugins list, not the full page data. Manual verification is enough; no specific perf test required.&#xA;&#xA;8. **Playwright e2e.**&#xA;   - Seeds a fixture with one outdated plugin.&#xA;   - Clicks Update.&#xA;   - Asserts the card enters the in-flight visual state and the action button is disabled.&#xA;   - Simulates the worker completing successfully.&#xA;   - Asserts within 2s the card shows `installed` and a newer `installedVersion`.&#xA;   - Runs a second scenario where the simulated worker fails; asserts the failure badge and that status stays `update-available`.&#xA;&#xA;9. **No regressions.** Install-from-registry flow (the ConfirmDialog path) adopts the same in-flight tracking; existing Playwright install flow still passes.
    </acceptance>
