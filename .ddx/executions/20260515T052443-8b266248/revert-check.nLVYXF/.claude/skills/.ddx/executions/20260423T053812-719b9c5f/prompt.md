## Observed / Motivation&#xA;&#xA;`.ddx/executions/&lt;ts&gt;-&lt;hash&gt;/{manifest.json, prompt.md, result.json}` bundles are already being produced per `ddx agent execute-bead` attempt — 469 of them on the current repo. They contain the canonical record of what an agent was asked to do and what verdict the attempt produced. **Nothing in the web UI surfaces them today.** A user who wants to ask &#34;why did this bead end up in this state?&#34; has to drop to the filesystem.&#xA;&#xA;Sessions (`sessions.jsonl` / sharded index per `ddx-2ceb02fa`) carry the harness/cost/token telemetry but not the verdict and not the bead linkage. Agent log streams (`agent-*.jsonl`) carry the tool-call trace but are scattered across hundreds of files. **Executions are the natural join** — they connect a bead, a session, a prompt, a verdict, and a tool-call stream.&#xA;&#xA;## Ground truth&#xA;&#xA;Per-execution on disk:&#xA;&#xA;```&#xA;.ddx/executions/&lt;ts&gt;-&lt;hash&gt;/&#xA;  manifest.json      { harness, model, base_rev, result_rev, verdict, bead_id, execution_dir }&#xA;  prompt.md          the prompt that was sent&#xA;  result.json        { verdict, rationale, ... }&#xA;```&#xA;&#xA;Relations:&#xA;- `manifest.bead_id` → bead&#xA;- a session id is recorded in bead events (`kind:cost`, `kind:routing`) and in the agent-log stream; both can map back to an execution via correlation metadata&#xA;- tool calls live in `.ddx/agent-logs/agent-*.jsonl` (and `agent-s-*.jsonl`), keyed by session id&#xA;&#xA;## Scope&#xA;&#xA;### Part 1 — GraphQL surface&#xA;&#xA;- `executions(projectId, first, after, beadId?, since?, until?, verdict?, harness?)` — paginated list of execution bundles.&#xA;- `execution(id)` — one bundle, fully materialized: manifest fields, prompt markdown, result JSON, linked bead (id + title), linked session (id + harness + model + cost + token totals), tool-call stream pointer.&#xA;- `executionToolCalls(id, first, after)` — paginated tool-call stream for the execution, read lazily from the agent-log file. Matches the event shape used by the workers detail page.&#xA;&#xA;Executions source of truth: directory scan of `.ddx/executions/` filtered/paginated at the resolver. Shard discipline: follow the sessions-index monthly shard pattern (`ddx-2ceb02fa`) if listing all executions grows beyond tens of thousands — per standing directive `ddx-9ce6842a`, raise the storage decision with the user before switching substrate.&#xA;&#xA;### Part 2 — Sidebar + routes&#xA;&#xA;- Add &#34;Executions&#34; to the project sidebar (`NavShell.svelte`).&#xA;- Routes:&#xA;  - `/nodes/&lt;n&gt;/projects/&lt;p&gt;/executions` — list view with filters (verdict, harness, bead, date range, free-text over prompt). Table columns: timestamp, bead, harness/model, verdict (pass/block/error), duration, cost.&#xA;  - `/nodes/&lt;n&gt;/projects/&lt;p&gt;/executions/&lt;id&gt;` — detail view with five tabs: **Manifest**, **Prompt** (rendered markdown), **Result** (rationale + raw JSON toggle), **Session** (harness, model, cost, token totals, link to `/sessions/&lt;id&gt;`), **Tool calls** (virtualized lazy-scroll list of tool_use + tool_result pairs with expandable payloads).&#xA;&#xA;### Part 3 — Cross-links (make the graph real)&#xA;&#xA;- **Bead detail:** new &#34;Executions&#34; section listing all executions for this bead with verdict + timestamp + link. Already have data: manifest.bead_id.&#xA;- **Session detail:** new &#34;Execution&#34; link pointing to the execution this session ran for, when the session has a linked execution.&#xA;- **Commits page:** already shows commits; if a commit was produced by an execution (result_rev == commit sha), surface the execution link.&#xA;- **Graph/Docs pages:** out of scope for this bead.&#xA;&#xA;### Part 4 — Perf&#xA;&#xA;- Fixture: ≥1,000 executions, per standing directive `ddx-9ce6842a`. Tool-call streams seeded for 10% of them.&#xA;- Targets: list view p95 ≤ 200ms HTTP, detail view without tool calls p95 ≤ 150ms HTTP, tool-call page-1 (first 50) p95 ≤ 300ms HTTP.&#xA;- If these can&#39;t be met with filesystem scan + in-memory filter, raise DB-substrate decision with user per standing directive.&#xA;&#xA;## Out of scope&#xA;&#xA;- Writing new execution bundles (that&#39;s already happening).&#xA;- Executions from other projects/nodes (keep per-project for v1).&#xA;- Tool-call payload redaction / privacy. File a follow-up if relevant.&#xA;- `.claude/worktrees/` worktree lifecycle tracking — covered by companion bead per user&#39;s split request.

**User story:** As a developer reviewing what my agents have done, I open the Executions view, filter to a specific bead or verdict or harness, click into one, and see the prompt it received, the verdict it produced, the session it corresponds to (with cost + tokens), and every tool call it made — all without leaving the browser.&#xA;&#xA;**Acceptance criteria:**&#xA;&#xA;1. **Sidebar entry.** &#34;Executions&#34; added to the project sidebar between &#34;Sessions&#34; and &#34;Personas&#34; (or wherever fits the existing visual order — design call; captured in the PR description). Active-highlight works.&#xA;&#xA;2. **List route.** `/executions` renders a paginated, filterable table with columns: timestamp, bead (id + title as link), harness/model, verdict (color-coded pass/block/error/other), duration, cost. Default sort: newest first. Filters: verdict, harness, bead, since/until date, free-text search over prompt first ~200 chars.&#xA;&#xA;3. **Detail route.** `/executions/&lt;id&gt;` shows five tabs as described (Manifest, Prompt, Result, Session, Tool calls). Tab switching doesn&#39;t re-fetch already-loaded data.&#xA;&#xA;4. **Tool calls pane.** Virtualized list. First 50 calls load with the page; subsequent pages load on scroll. Each entry shows tool name + collapsed input/output, expandable on click. Links to the raw agent-log file for operators who want to diff.&#xA;&#xA;5. **GraphQL completeness.** `executions`, `execution`, `executionToolCalls` resolvers exist and pass feat-008-style integration tests. Schema documented in `schema.graphql`.&#xA;&#xA;6. **Cross-links present.**&#xA;   - Bead detail renders an &#34;Executions&#34; section listing linked executions with verdict + timestamp, each clicking into the detail route.&#xA;   - Session detail shows &#34;Execution&#34; link when the session has one.&#xA;   - Commits page shows execution link when commit sha == execution.result_rev.&#xA;&#xA;7. **Fixture + perf.**&#xA;   - Seeded fixture: ≥1,000 executions, ≥100 with seeded tool-call streams (≥50 calls each).&#xA;   - List view on this fixture: p95 ≤ 200ms HTTP.&#xA;   - Detail view (no tool calls): p95 ≤ 150ms HTTP.&#xA;   - Tool-call page 1 (first 50 entries): p95 ≤ 300ms HTTP.&#xA;   - If any target cannot be met in-place, raise DB-substrate decision with user before switching.&#xA;&#xA;8. **Playwright e2e.**&#xA;   - Seeds a project with 3 executions across 2 beads with mixed verdicts.&#xA;   - Navigates to `/executions`, asserts the table renders 3 rows and filter-by-verdict narrows correctly.&#xA;   - Clicks into one execution, asserts all 5 tabs exist and the Prompt tab shows the seeded prompt.&#xA;   - Opens the Tool calls tab, expands an entry, asserts input/output rendering.&#xA;   - Clicks the linked bead from the detail, asserts navigation to bead detail and that the Executions section on that bead lists this execution.&#xA;&#xA;9. **Depends on.**&#xA;   - Soft dependency on `ddx-2ceb02fa` for the Session tab&#39;s cost/token numbers. If not yet landed, render &#34;not available&#34; for those fields and wire them in after.&#xA;   - Adjacent: `ddx-6904a90b` (billing mode) — Session tab&#39;s cost shows the correct bucket once that lands.
