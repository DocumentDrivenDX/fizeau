    <title>Bead list and cross-project all-beads are too slow: establish a GraphQL perf harness and fix N·M scan pattern</title>
    <description>
## Observed&#xA;&#xA;Per-project `/beads` page is noticeably slow; the cross-project &#34;all beads&#34; view is substantially slower. Related: `ddx-ad0db8fd` already identified the single-bead-detail slowness caused by the same storage pattern.&#xA;&#xA;## Root cause (same pattern, different call site)&#xA;&#xA;`cli/internal/server/graphql/resolver_beads.go:22,40` —&#xA;&#xA;- `Beads` (cross-project) and `BeadsByProject` (single-project) both call `r.State.GetBeadSnapshots(status, label, projectID, search)`.&#xA;- `GetBeadSnapshots` (`cli/internal/server/state_graphql.go:49-105`) iterates every registered project, constructs a `bead.Store` per project, calls `ReadAll()` (scan + JSONL parse of every bead file), filters in Go. Cost is `O(projects × beads_per_project)` on every call, independent of the filter/page size.&#xA;- All filtering (status, label, project, search) is Go-side, post-read. The page-size window is applied in `beadConnectionFromSnapshots` after the full corpus is already materialized.&#xA;- Cross-project &#34;all beads&#34; = same loop, just without the projectID filter. It is the worst case by design.&#xA;&#xA;This is the same shape that&#39;s breaking the Sessions feed (`ddx-2ceb02fa`) and single-bead detail (`ddx-ad0db8fd`). A point fix here is insufficient — we need a measurement surface first so we stop relying on anecdote.&#xA;&#xA;## Proposed direction&#xA;&#xA;### Part 1 — Perf harness (foundational; enables everything else)&#xA;&#xA;Introduce a reusable GraphQL perf harness under `cli/internal/server/perf/` or `cli/internal/server/graphql/bench_test.go`:&#xA;&#xA;- Seeded fixture builder: generates N projects × M beads with configurable distribution of status/labels/dependencies. Reuses existing store primitives (no fake data layer).&#xA;- Go benchmark + scriptable HTTP runner (wrapper around an already-running server) reporting p50/p95/p99 per query shape.&#xA;- Targets covered on day one: `bead(id:)`, `beadsByProject(projectID, first, after, status, label, search)`, `beads(projectID=nil, first, after, status, label)`, `docGraph`, `sessions` (so the sessions shard work `ddx-2ceb02fa` inherits this harness). New query targets are added by writing a single table row.&#xA;- Thresholds are **recorded, not asserted**, in the first pass. CI surfaces a regression if a threshold is breached beyond a configured delta (e.g., +25% p95). Initial pass is &#34;land the harness and record baseline&#34;; a follow-up wires it into CI.&#xA;&#xA;### Part 2 — Fix the scan pattern&#xA;&#xA;Minimum viable, in priority order:&#xA;&#xA;1. **Scope by project when the query is scoped.** `BeadsByProject` should call a new `GetBeadSnapshotsForProject(projectID, filters…)` that only opens that one project&#39;s store. Trivial change; eliminates the cross-project factor for the common case.&#xA;2. **Early filter pushdown.** `GetBeadSnapshots` should apply status/label/projectID filters at the file-parse boundary where cheap, not after materializing every bead. (Still reads every file — deeper fix below.)&#xA;3. **Lightweight in-memory index with invalidation.** After (1) and (2), if &#34;all beads&#34; or search is still slow, introduce an index keyed `{projectID, beadID}` with basic metadata (status, labels, title, updatedAt). Refresh on bead write + debounced filesystem watch. Full-body reads stay lazy — the index never holds bodies. This is the same storage discipline we adopted for sessions in `ddx-2ceb02fa`.&#xA;&#xA;The harness from Part 1 is the gate for choosing between (2) and (3); don&#39;t commit to the index without numbers.&#xA;&#xA;## Out of scope&#xA;&#xA;- Full-text search infrastructure (tracked separately if needed).&#xA;- Changing the on-disk bead format.&#xA;- Cross-node federation performance. Current concern is a single server reading a single workspace.
    </description>
    <acceptance>
**User story:** As a developer using the web UI on a real-world-sized workspace (tens of projects × hundreds of beads each), the beads list opens in under a second and the cross-project all-beads view stays usable. Before any perf fix lands, we can reproduce the problem and measure improvement — not guess.&#xA;&#xA;**Acceptance criteria:**&#xA;&#xA;1. **Perf harness exists and is reusable.**&#xA;   - Fixture builder generates `N × M` beads across projects with configurable shape (status mix, label density, dep count). Reuses the real `bead.Store` write path (no fake writer).&#xA;   - Benchmark covers at minimum: `bead(id:)`, `beadsByProject`, `beads` (cross-project), `docGraph`, `sessions`. Adding a new target is a single-table-row change.&#xA;   - Each target produces p50/p95/p99 for in-process calls and end-to-end HTTP. Results write to a human-readable report (markdown or JSON) plus a machine-readable artifact suitable for CI diffing later.&#xA;&#xA;2. **Baseline recorded.** First run of the harness on the current `main` binary is stored in the bead notes and in `docs/helix/04-observe/perf/YYYY-MM-DD-baseline.md` (or equivalent existing location). Numbers include hardware + go version footer.&#xA;&#xA;3. **BeadsByProject scope fix.** `BeadsByProject` calls a new `GetBeadSnapshotsForProject(projectID, …)` that opens exactly one project&#39;s store. Unit test asserts only that project&#39;s store is opened (no iteration over other projects).&#xA;&#xA;4. **Filter pushdown.** Status/label/projectID filters apply at the per-file / per-entry parse boundary, not post-materialization. Benchmark captures before/after for the common filtered case (e.g., `status=open`).&#xA;&#xA;5. **Perf targets on fixture (10 projects × 500 beads = 5000 beads total):**&#xA;   - `bead(id:)` p95 ≤ 50ms in-process / ≤ 200ms HTTP (already covered by `ddx-ad0db8fd`; this bead may inherit the work).&#xA;   - `beadsByProject(first=50)` p95 ≤ 75ms in-process / ≤ 250ms HTTP.&#xA;   - `beads(first=50, no projectID)` p95 ≤ 200ms in-process / ≤ 500ms HTTP.&#xA;   These are targets for the **first** pass (scope + pushdown). If they are not met with (1)+(2), escalate to the in-memory index (step 3 of Part 2 in the description) and update these targets with the new baseline.&#xA;&#xA;6. **Stale-read safety.** Regardless of fix chosen, a test writes a bead via `ddx bead update`, then queries it via GraphQL, then asserts the new value is visible on the very next request. Works under concurrency: N concurrent readers + writers interleaved without torn reads or panics.&#xA;&#xA;7. **CI posture (soft).** The harness is runnable via `make bench-graphql` (or the project&#39;s equivalent). CI wiring to fail on &gt;25% regression is a follow-up bead — recorded in this bead&#39;s notes with the proposed threshold but not implemented here.&#xA;&#xA;8. **Playwright smoke.** Opening `/beads` on the fixture project returns an interactive (clickable) list within 1s. Opening the cross-project `/beads` page returns within 2s. These are ceilings, not p95 targets — guards against outright regression, not fine-grained perf.&#xA;&#xA;9. **Cross-reference.** This bead&#39;s notes list `ddx-ad0db8fd` (bead detail) and `ddx-2ceb02fa` (sessions shard/index) as the other two bead/session perf threads that should inherit this harness.
    </acceptance>
