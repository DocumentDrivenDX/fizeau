    <title>Server worker record Attempts/Successes/Failures stuck at 0 until loop exits</title>
    <description>
ddx server workers show &lt;id&gt; reports Attempts: 0  Success: 0  Failed: 0 for a worker that has been running for 30+ minutes and has visibly completed multiple beads (per the worker log stream and per landed commits in main). The counters only update at the end of worker.Run(), when the loop exits. While the loop is mid-flight the server WorkerRecord stays at zero, making the show endpoint useless for live progress.&#xA;&#xA;Root cause: in cli/internal/server/workers.go startWorker(), the call site is:&#xA;&#xA;    loopResult, err := worker.Run(ctx, ...)  // BLOCKING — only returns when loop exits&#xA;    // THEN updates record.Attempts = loopResult.Attempts&#xA;&#xA;The ExecuteBeadLoopResult is built up in a local variable inside the loop Run() (see cli/internal/agent/execute_bead_loop.go line 147 &#39;result := &amp;ExecuteBeadLoopResult{}&#39;, incremented at line 255 &#39;result.Attempts++&#39; etc.) but the server holds no reference to it during execution.&#xA;&#xA;Fix options:&#xA;&#xA;A) ExecuteBeadLoopOptions gains a Progress field of type pointer to ExecuteBeadLoopResult. The caller pre-allocates one, passes it in, and the loop writes to it under a sync.Mutex. The server reads from the same pointer (with the mutex) when serving GET /api/agent/workers/{id}.&#xA;&#xA;B) ExecuteBeadLoopOptions gains an OnBeadResult callback that fires after each bead. The server subscribes and updates record.Attempts/Successes/Failures incrementally.&#xA;&#xA;C) The server already passes an EventSink and the loop already emits bead.result events to it. The server can subscribe to its own sink (parse the JSONL events) and update the counters from those events.&#xA;&#xA;Option A is simplest and least invasive. Option C is cleanest because it reuses the existing event stream.
    </description>
    <acceptance>
Backend Go test in cli/internal/server/workers_test.go starts a worker against a fake executor that completes 3 beads (2 success, 1 failure) with a 50ms delay between each. Polls Server.workers.Show(id) every 30ms and asserts that BEFORE the loop exits, at least one Show response reports Attempts &gt;= 1. Final Show after exit reports Attempts=3, Successes=2, Failures=1. Smoke acceptance: with a running worker, ddx server workers show {id} reports a non-zero Attempts count within 30 seconds of the first bead completing, not after the entire loop exits.
    </acceptance>
