    <title>ddx doctor should verify running binary is in canonical install location</title>
    <description>
install.sh installs the ddx binary to $HOME/.local/bin/ddx as the canonical location. But users can end up with multiple copies (e.g. /home/erik/bin/ddx from a stale install, /usr/local/bin/ddx, or a dev build on PATH ahead of the canonical one). This session had a real incident: /home/erik/bin/ddx was 053171bf (older) while /home/erik/.local/bin/ddx was 40f8db0d (current) — the shell picked the wrong one for hours, causing confusion about whether blocker fixes were actually active. ddx doctor should detect mismatches between the running binary&#39;s path and the canonical install location AND warn when multiple ddx copies exist on PATH with different SHAs.
    </description>
    <acceptance>
ddx doctor (1) reports the absolute path of the currently-executing binary via os.Executable(); (2) checks whether that path matches $HOME/.local/bin/ddx (the install.sh canonical location); (3) walks PATH and reports every other &#39;ddx&#39; found, flagging any whose SHA-256 differs from the running binary as a warning; (4) suggests &#39;rm &lt;stale&gt; &amp;&amp; cp &lt;current&gt; &lt;stale&gt;&#39; or &#39;re-run install.sh&#39; to fix. Test covers the multi-binary-on-PATH scenario by shadowing a fake ddx in a temp dir earlier on PATH.
    </acceptance>
