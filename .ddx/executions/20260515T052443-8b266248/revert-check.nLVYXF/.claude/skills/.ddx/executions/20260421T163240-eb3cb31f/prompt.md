    <description>
## Target UX&#xA;&#xA;The user-facing profile vocabulary (cheap / default / fast / smart) should map to concrete routing policies that take full advantage of local models without blocking on their limits. Specifically:&#xA;&#xA;### default profile (the common case)&#xA;Try local models first, escalate to cloud if:&#xA;- local model&#39;s context window is too small for the bead prompt&#xA;- local model lacks a capability the bead requires (tools, structured output, etc.)&#xA;- local dispatch fails (provider unreachable, timeout, model-side error)&#xA;- bead reopens due to post-run check failure (review-malfunction, land-conflict, etc.)&#xA;&#xA;Escalation target: next tier up in the ladder (standard cloud — e.g. codex gpt-5.4 within subscription). If THAT fails, escalate once more to smart cloud. Then stop and unclaim.&#xA;&#xA;### cheap profile&#xA;Local only. Never escalate. Intended for high-volume exploratory work where cost dominates and adequate-quality is sufficient. If the cheap tier can&#39;t serve the bead, the bead unclaims and waits.&#xA;&#xA;### fast profile&#xA;Cloud-first, skip local entirely. Intended for latency-critical or interactive work where local inference&#39;s 9s warmup is unacceptable. Starts at a fast cloud model (e.g. gemini-flash or haiku) and escalates to smart on failure.&#xA;&#xA;### smart profile&#xA;Cloud-high-quality from the start. Intended for bead-review, adversarial checks, or beads whose AC demands high-quality reasoning. No escalation — if smart fails, the work fails meaningfully.&#xA;&#xA;## Why this matters (dogfooding signal)&#xA;&#xA;As of 2026-04-18, DDx&#39;s default is cheap = qwen/qwen3.6 local (vidar-omlx). With v0.3.14 shipping the SSE-comment-frame filter, the local path actually works end-to-end. The queue can drain mostly on local hardware for pennies per bead, escalating only when the bead genuinely needs cloud quality. That&#39;s the throughput-per-dollar goal (goal:throughput-per-dollar label).&#xA;&#xA;## Prereqs (already filed under ddx-2d974641 or sibling beads)&#xA;&#xA;This epic is blocked on the autorouting epic delivering its exit-gate UX — &#39;declare provider + use canonical ref + autorouting works&#39;. Specifically:&#xA;&#xA;- ddx-8610020e (RouteRequest.Provider field) — foundation&#xA;- ddx-0486e601 (fuzzy normalization + per-provider) — lets cheap-tier model refs resolve to provider-native IDs&#xA;- ddx-2f5a2284 (catalog/Profile discovery integration) — lets Profile tier overrides retarget to provider-native IDs&#xA;- ddx-3c5ba7cc (tier escalation --provider affinity) — the lower half of this epic&#39;s &#39;escalate on failure&#39; semantics&#xA;- ddx-4817edfd (capability gating) — prevents routing to a model that can&#39;t serve the bead&#xA;&#xA;With those in place, the profile-level policy here is mostly CONFIGURATION + a small amount of scoring-layer logic.&#xA;&#xA;## Scope&#xA;&#xA;### Config schema additions&#xA;&#xA;.ddx/config.yaml agent.routing extends with an explicit per-profile ladder:&#xA;&#xA;    agent:&#xA;      routing:&#xA;        profile_ladders:&#xA;          default: [cheap, standard, smart]      # local first, escalate&#xA;          cheap:   [cheap]                        # local only&#xA;          fast:    [fast, smart]                  # cloud fast, escalate to smart&#xA;          smart:   [smart]                        # cloud high-quality, no escalation&#xA;        model_overrides:&#xA;          cheap:    qwen/qwen3.6&#xA;          fast:     kimi/k2.5&#xA;          standard: codex/gpt-5.4                # new: subscription cloud, mid-quality&#xA;          smart:    minimax/minimax-m2.7&#xA;&#xA;Current profile_priority field (a flat list) becomes a legacy alias for profile_ladders.default with a deprecation warning.&#xA;&#xA;### Tier-escalation policy (routing.go)&#xA;&#xA;For a given profile P, iterate profile_ladders[P]:&#xA;- Evaluate model_overrides[tier] resolves on a viable candidate (with --provider affinity from ddx-3c5ba7cc).&#xA;- If yes, try it; on failure (execution_failed / land_conflict / post_run_check_failed), move to next tier.&#xA;- Record tier decision + escalation chain in routing event on the bead so telemetry reflects where work actually landed.&#xA;&#xA;### User-facing CLI&#xA;&#xA;- ddx work --profile default — the new primary invocation. Most users run this.&#xA;- ddx work --profile cheap / fast / smart — opt-in deviations.&#xA;- --max-tier continues to cap the ladder (explicit user override beats config).&#xA;&#xA;### Telemetry + measurement&#xA;&#xA;Every execution event records: requested_profile, requested_tier, resolved_tier, escalation_count, final_tier. Aggregated metrics (FEAT-016 / ddx-4b59b816) report per-profile throughput, per-tier cost, per-bead escalation rate.&#xA;&#xA;## Exit gate&#xA;&#xA;&#39;ddx work --profile default --once --local&#39; against a ready bead on a host with a healthy local provider AND healthy cloud fallback:&#xA;- Claims the bead.&#xA;- Tries cheap tier (local). If it succeeds, closes bead with tier=cheap recorded.&#xA;- If cheap fails for any escalation-triggering reason, automatically retries at standard tier without operator intervention.&#xA;- If standard fails, retries at smart.&#xA;- Final outcome recorded faithfully (success at whatever tier landed, or exhaustion if all tiers failed).&#xA;&#xA;## Out of scope&#xA;&#xA;- Dynamic profile selection (&#39;auto-pick profile based on bead complexity&#39;). Too much magic; explicit profile choice is fine.&#xA;- Cost budgets / circuit breakers. FEAT-014 (token awareness) tracks this separately.&#xA;- Per-bead profile overrides (bead-level &#39;requires-profile: smart&#39;).
