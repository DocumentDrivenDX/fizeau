    <title>routing: proactive endpoint aliveness probing and durable cache for local providers</title>
    <description>
PROBLEM&#xA;Fizeau has no proactive aliveness check for configured providers. A dead local provider (e.g. bragi:1234, grendel:1234) is only learned to be dead by attempting a request against it and eating the dial timeout. The failure feedback is stored in internal/routehealth.Store, which is:&#xA;1. Process-local (sync.RWMutex map; not persisted to disk).&#xA;2. Short-TTL (DefaultCooldown = 30s; entries expire and are re-tried).&#xA;&#xA;Result for an operator with bragi unplugged but configured:&#xA;- Every fresh `ddx work` or `ddx try` process re-probes bragi on its first routing decision, eating a 5s dial timeout (post fizeau-b5757624) or 30s pre-fix.&#xA;- Even within a long-running `ddx work --watch`, the route exclusion expires every 30s and bragi gets re-attempted on the next cycle.&#xA;- Aggregate cost over a 27-bead drain on 2026-05-14: minutes of dial timeouts and routing thrash, leaking into operator-visible queue stalls.&#xA;&#xA;A quota-recovery probe loop exists at service_probe.go for providers marked `quota_exhausted` (periodic, with retry-after backoff). It does NOT cover `provider_connectivity` failures — those have no proactive recovery path.&#xA;&#xA;ROOT CAUSE&#xA;- internal/routehealth/store.go owns failure tracking; it is in-memory only, with no persistence interface (verified by file listing: only store.go / lease_store.go / utilization_store.go — no persist.go).&#xA;- service_probe.go&#39;s QuotaRecoveryProber only handles ProviderQuotaStateStore — not RouteHealth failures.&#xA;- No service-startup probe for configured providers. Providers in fizeau config are trusted to be reachable until a request proves otherwise.&#xA;&#xA;PROPOSED FIX&#xA;1. Add a startup aliveness probe to internal/provider/openai/reachability.go (or a new internal/routehealth/probe.go). For each configured provider with a non-cloud base_url, run a single TCP-connect probe (bounded ~2s) on Service.New(). Record the result in routehealth.Store via a new RecordProbe(key, success, observedAt) method.&#xA;2. Add a periodic background probe loop (mirroring runQuotaRecoveryProbeLoop in service_probe.go): re-probe providers whose last probe is older than HealthProbeInterval (default 60s). Bound to local-ish providers — skip the cloud subscription routes that handle their own health.&#xA;3. Make routehealth.Store optionally persistent. ServiceOptions.PersistRouteHealth path (already proposed in fizeau-7cd66e6b) writes the probe + failure cache to a JSON snapshot with HealthSignalTTL (default 10 min). Subsequent processes read the snapshot at startup so a fresh ddx try doesn&#39;t re-discover bragi-is-dead.&#xA;4. Extend the routing gate: a candidate whose endpoint has a Probe.Success=false within HealthSignalTTL is FILTERED (not just score-demoted) when alternates exist. New FilterReason: endpoint_unreachable.&#xA;5. RouteDecision evidence: include the probe state (last_probe_at, last_probe_success) for each candidate so operators can audit why a candidate was filtered.&#xA;&#xA;NON-SCOPE&#xA;- Don&#39;t probe cloud subscription routes (claude, openrouter, etc.) — those have their own health story via quota probing.&#xA;- Don&#39;t change the connection-failure retry policy (fizeau-8134d6db already shipped: 1 attempt, fail fast).&#xA;- Don&#39;t introduce a daemon — the periodic probe runs inside Service goroutines, same lifetime as quota-recovery loop.&#xA;- Don&#39;t probe at request time — request-time dial is the existing reactive path; this bead adds proactive probing on a separate schedule.&#xA;&#xA;DEPS&#xA;- Composes with fizeau-7cd66e6b (persist route-health across processes) — same persistence layer.&#xA;- Composes with fizeau-361f4f4b (router reliability epic) — provides the missing aliveness signal that epic&#39;s AC #1 (&#34;fall through to subscription within one decision&#34;) depends on.
