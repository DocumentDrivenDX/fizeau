    <title>bench: classifier misses retry-spam cells (turns inflated by fiz internal retries on connection errors)</title>
    <description>
fiz retries transient provider errors (connection refused, EOF, 5xx) internally up to ~5 times with exponential backoff. Each retry attempt counts as a turn in the matrix, inflating the turns counter even when zero useful work happened. Result: cells where the provider was completely unreachable end up with turns=6 (5 retries), output_tokens=0, terminated_mid_work=nil, but `matrixHasMeaningfulAttempt(report)` returns true because turns &gt; 0 — bypassing the new invalid_provider classifier rule introduced in 6688326e (fizeau-e874633d).&#xA;&#xA;Concrete incident: on 2026-05-11 ~15:25Z, vidar:1235 oMLX server crashed. 70 cells silently failed before detection. fiz.txt showed:&#xA;  2026/05/11 15:29:41 WARN provider error, retrying attempt=1 err=&#34;unexpected EOF&#34;&#xA;  2026/05/11 15:29:42 WARN provider error, retrying attempt=2 err=&#34;connect: connection refused&#34;&#xA;  ... etc through attempt=5&#xA;Each cell ended with turns=6, wall=60-200s, final_status=graded_fail. Classifier left them as plain graded_fail — i.e. &#34;the model attempted and failed&#34; — polluting vidar&#39;s pass-rate denominator with what was actually 100% provider unavailability.&#xA;&#xA;The hole is structural: hasMeaningfulAttempt currently looks at turns/tokens/tool_calls &gt; 0. fiz&#39;s retry counter is one of the inputs to that signal, but a retry isn&#39;t a meaningful attempt — it&#39;s the absence of one.&#xA;&#xA;PROPOSED FIX&#xA;&#xA;Add a stronger &#34;no real attempt&#34; signal that distinguishes retry-spam from real turns. Two complementary options:&#xA;&#xA;A. **fiz records retry attempts separately** from successful llm.request/response pairs. The session jsonl would carry an `llm.retry` event (or annotate llm.request with an `attempt` field). The matrix runner already reads the session jsonl in readSessionSignalsFromCell — it can extend the count to distinguish &#34;had_completed_response&#34; from &#34;had_request&#34;. Then: graded_fail + !had_completed_response + had_request → invalid_provider, regardless of turns count.&#xA;&#xA;B. **Token-based fallback in the existing classifier**: even if turns &gt; 0, if input_tokens=0 AND output_tokens=0 AND had_llm_request=true AND tmw=nil, classify as invalid_provider. The intuition: a real turn consumes prompt tokens (input_tokens &gt; 0 from session.end usage). All-zero token counts means no completed round-trip ever happened. Cheaper than (A) but a bit more heuristic.&#xA;&#xA;In-scope files for Part B (simpler, ships first):&#xA;- cmd/bench/matrix_invalid.go — extend the case &#34;graded_fail&#34; branch with the token-based check before the !hasAttempt branch.&#xA;- cmd/bench/matrix_invalid_test.go — fixture: graded-fail-retry-spam.json with turns=6, input_tokens=0, output_tokens=0, had_llm_request=true, tmw=nil → expects invalid_provider.&#xA;- scripts/benchmark/reclassify-invalid.py — mirror.&#xA;- Run reclassifier; expect ~70+ historical vidar cells from 2026-05-11 to flip invalid_setup/empty → invalid_provider.&#xA;&#xA;In-scope for Part A (deeper, file as separate bead if simple fix isn&#39;t enough):&#xA;- internal/core/loop.go — distinguish &#34;request attempted but provider error retried&#34; from &#34;completed request/response pair&#34; in event emission. Add an `attempt` field on llm.request or a new event type.&#xA;- scripts/benchmark/harbor_agent.py FizeauAgent.parse_telemetry — count completed responses (not just requests) when populating turns/tokens fields.&#xA;- cmd/bench/matrix.go readSessionSignalsFromCell — return n_completed_responses alongside the existing tmw/had_llm_request signals.&#xA;&#xA;ACCEPTANCE IS A COMMAND&#xA;&#xA;1. go test ./cmd/bench/... -run TestClassifyMatrixInvalid passes with the new fixture graded-fail-retry-spam.json asserting invalid_provider.&#xA;2. .venv-report/bin/python scripts/benchmark/reclassify-invalid.py --dry-run flags the cells from the 2026-05-11 vidar incident (70+ cells) as transitioning to invalid_provider.&#xA;3. Apply pass leaves those cells excluded from vidar&#39;s pass-rate denominator and tagged for --retry-invalid reruns.&#xA;&#xA;WHY THIS MATTERS&#xA;&#xA;Without this, every server-side outage (oMLX crash, vLLM hang, network blip) gets attributed to model quality instead of provider availability.
