PROBLEM&#xA;Under policy=default (auto-route mode, no --harness/--provider/--model pin), Fizeau collapses each subprocess harness to ONE scoring candidate — its DefaultModel. Multi-tier subscription harnesses like claude (which advertises sonnet/opus/haiku as siblings under one auth) therefore only ever offer their default tier to the auto-route scorer. Sonnet and haiku can never win on cost vs opus even though the catalog has full power/cost metadata for all three.&#xA;&#xA;Concrete symptom from downstream DDx caller: every policy=default dispatch lands on claude/opus-4.7 because internal/harnesses/registry.go:51 sets DefaultModel=&#34;opus-4.7&#34;. For a 2207-token implementation prompt at role=implementer, opus won score 100.5. When the same dispatch is rerun with --model sonnet-4.6 pinned, sonnet scores 196.0 — it would have beaten opus by 95.5 points if it had been enumerated.&#xA;&#xA;Cost gap is material: opus is 5x sonnet on input tokens, ~5x on output (cost_input_per_m: 15.0 vs 3.0; cost_output_per_m: 75.0 vs 15.0 per internal/modelcatalog/catalog/models.yaml). A project running ddx work continuously is defaulting every dispatch to the most expensive tier.&#xA;&#xA;ROOT CAUSE (tentative — pointers, not patch)&#xA;service_routing.go:1172-1189 (auto-route candidate add loop):&#xA;&#xA;    for _, h := range entries {&#xA;        if h.DefaultModel != &#34;&#34; {&#xA;            add(h.DefaultModel, true, status)      // adds opus-4.7 for claude&#xA;        }&#xA;        for _, modelID := range h.SupportedModels {&#xA;            add(modelID, true, status)              // SHOULD add sonnet-4.6 too&#xA;        }&#xA;        ...&#xA;    }&#xA;&#xA;subprocessHarnessModelIDs(&#34;claude&#34;, cfg) at service_models.go:91-101 returns the full [&#34;sonnet&#34;, &#34;sonnet-4.6&#34;, &#34;opus&#34;, &#34;opus-4.7&#34;, &#34;claude-sonnet-4-6&#34;] set.
