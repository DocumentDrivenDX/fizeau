## 1. Problem Statement and User Impact

DDx aims to be the shared infrastructure platform for document-driven development
— providing the tools that developers and workflow systems use to manage the
documents AI agents consume to build software. The repository currently has an
asymmetric maturity profile that limits DDx's ability to serve this mission.

### What is mature and stable

| Area | Feature | Status |
|------|---------|--------|
| CLI surface | FEAT-001 | 60+ commands across 12 domains |
| Bead tracker | FEAT-004 | Complete: CRUD, claims, evidence, schema validation, bd/br compat |
| Artifact conventions | FEAT-005 | `ddx:` frontmatter, graph inclusion, type conventions |
| Document dependency graph | FEAT-007 | Core engine: parse, graph, stale, cascade, stamp |
| Agent service | FEAT-006 | Invocation, quorum, token extraction, session logging (metadata) |
| Execution substrate | FEAT-010 | Definitions, runs, metric projection, bead-backed collections |
| Token awareness | FEAT-014 | In progress: codex/claude parsing, usage command |

### What needs significant work

| Area | Feature | Gap |
|------|---------|-----|
| Server & MCP | FEAT-002 | Routes exist but contract coverage incomplete; missing API envelope, auth, most endpoints |
| Web UI | FEAT-008 | In-progress tracker item; framework decision made (ADR-002/005) but no code |
| Registry | FEAT-009 | In-progress tracker item; `ddx install/search/verify` not yet functional |
| Agent skills | FEAT-011 | Not started; SKILL.md format defined but no skills written |
| Git awareness | FEAT-012 | Not started; auto-commit, history queries, checkpoints |
| Multi-agent coordination | FEAT-013 | Framing only; claim identity extension, MCP write tools, worktree dispatch |
| Release readiness | SD-007 | CI pipeline, demo automation, Pages deployment, goreleaser |

### Residual cleanup

| Area | Issue |
|------|-------|
| Exec model reconciliation | Epic ddx-facb7aa5 open; most child tasks closed |
| Agent session body capture | Metadata logged but prompt/response/log bodies not yet captured per TD-006 |

### Who benefits

1. **Developers using AI agents** — need DDx server + MCP to give agents
   programmatic access to project documents, beads, and execution evidence
   instead of manual context assembly.
2. **Workflow tool authors** (HELIX is the first) — need DDx registry to
   distribute workflow packages, DDx server for remote observation, and DDx
   multi-agent primitives for concurrent agent coordination.
3. **AI agents (machine consumers)** — need MCP tools to browse document
   libraries, inspect bead queues, and query execution history without relying
   on humans to copy-paste context.

### What happens if we don't act

DDx remains a CLI-only tool. Workflow tools like HELIX continue using DDx beads
and agent dispatch via filesystem-only interfaces. Network-accessible document
browsing, remote agent observation, package distribution, and multi-agent
coordination remain unavailable. DDx cannot fulfill its stated vision of being
"what npm is for packages" for document-driven development.

## 2. Requirements Analysis

### 2.1 Functional Requirements

#### Server & MCP (FEAT-002) — P0

- Complete REST API coverage with standardized response/error envelope.
- Endpoints: documents (list, read, search, persona), beads (list, show,
  ready, blocked, status, dep tree, create, update, claim, unclaim), doc graph
  (graph, stale, show, deps, dependents), agent sessions (list, detail),
  executions (definitions, runs, detail, history), registry (search, installed,
  verify, install, uninstall), health (liveness, readiness).
- MCP tool parity with all read endpoints; write tools for beads (FEAT-013).
- Document write + auto-commit endpoint (requires FEAT-012).
- API key middleware for non-localhost binds.
- CORS restricted to configured origins.

#### Web UI (FEAT-008) — P1

- Embedded SPA served from Go binary via `embed.FS` (per ADR-002).
- Document browser: list by category, full-text search, markdown rendering with
  syntax highlighting, frontmatter metadata, clickable dependency links.
- Bead master-detail list: searchable, sortable, filterable by
  status/priority/label/owner. Kanban board with drag-and-drop status changes.
- Dependency graph visualization via D3.js or Cytoscape.js: color-coded
  staleness, zoom/pan, node filtering.
- Agent log monitor: live-updating list, expandable session detail.
- Status dashboard: summary cards, quick links.
- Client-side data layer: plain JS arrays + MiniSearch per ADR-005 v2 (~10,000 bead cap before
  server-side pagination needed).

#### Registry & Package Install (FEAT-009) — P0

- `ddx install <package>` — shallow clone, copy to targets, record in lockfile.
- `ddx search <query>` — search registry index.
- `ddx installed` — list installed packages with versions.
- `ddx uninstall <package>` — remove package and lockfile entry.
- `ddx verify` — check lockfile integrity via tree-hash and per-file hashes.
- `.ddx/lock.yaml` with commit SHA + tree hash per ADR-003.
- Plugin cache population at `~/.cache/ddx/library/plugins/`.
- Registry cache with configurable TTL at `~/.cache/ddx/registries/`.
- Install lock mechanism (parallel to bead lock) to prevent concurrent installs.

#### Agent Skills (FEAT-011) — P1

- `/ddx-bead` — guided bead creation and triage.
- `/ddx-agent` — guided agent dispatch with harness/model suggestions.
- `/ddx-install` — guided package installation.
- `/ddx-status` — project health overview.
- SKILL.md format with `name`, `description`, `argument-hint` frontmatter.
- Installation via `ddx init` (symlink/copy to `~/.agents/skills/ddx-*`)
  and updates via `ddx install ddx-skills`.

#### Git Awareness (FEAT-012) — P1

- Auto-commit option: `always | prompt | never` per `git.auto_commit` config.
- Structured commit messages: `docs(<artifact-id>): <description> [ddx: <op>]`.
- `ddx doc history <id>` — git log filtered by artifact path.
- `ddx doc diff <id> [refs]` — diff between revisions.
- `ddx doc changed [--since ref]` — documents changed since ref.
- `ddx checkpoint <name>` — create `ddx/` prefixed git tag.
- MCP tools: `ddx_doc_history`, `ddx_doc_diff`, `ddx_doc_changed`.
- Graceful degradation: features disabled outside git repos with clear message.
- Exclusive commit lock for concurrent multi-agent auto-commits (FEAT-013
  interaction).

#### Multi-Agent Coordination (FEAT-013) — P1

- Extend bead claim metadata with machine identifier and session ID (not just
  PID).
- MCP bead write tools: `ddx_bead_create`, `ddx_bead_update`,
  `ddx_bead_claim`.
- Worktree-aware agent dispatch: `ddx agent run --worktree <name>`, claim
  records branch.
- Remote worker pattern: coordinator publishes work package, worker picks up
  via branch push + bead update (future phase).

#### Release Readiness (SD-007)

- CI pipeline: `lefthook run ci` on every push and PR via GitHub Actions.
- Demo automation: deterministic scripts in `scripts/demos/`, asciinema
  recording + agg GIF rendering.
- GitHub Pages deployment gated on CI passing.
- Multi-platform release builds via goreleaser with changelog generation.
- E2E smoke tests (TP-007) in CI.
- Semantic versioning tied to git tags; `ddx version` prints tag.

#### Platform Cleanup

- Close exec model reconciliation epic (ddx-facb7aa5).
- Implement agent session body capture per TD-006.
- Complete FEAT-014 token awareness (codex/claude parsing, usage command).

### 2.3 Constraints

- Go for CLI and server; TypeScript SPA for web UI (ADR-001, ADR-002).
- File-backed storage, no external database for v1.
- Localhost default for server; optional API key for remote (FEAT-002).
- MIT license, open source.
- Agent permissions default to safe mode (FEAT-006).
- Config changes are additive: existing `.ddx/config.yaml` files work unchanged
  with new sections; unknown keys preserved.

### 2.4 Open Questions

1. **API key scope (FEAT-002):** Should API key be required for all endpoints
   when non-localhost, or only mutations and sensitive reads? *Recommendation:*
   all endpoints uniformly — simpler to reason about.
2. **Server-side pagination timing:** Required from day 1 for large repos, or
   deferred? *Recommendation:* offset/limit with max page size (1000) from
   day 1; total count in response meta. Plain JS arrays + MiniSearch handle bead
   filtering per ADR-005.
3. **Registry first-fetch trust (ADR-003):** Interactive TOFU prompt or silent
   with warning? *Recommendation:* silent with warning on first fetch; `ddx
   verify` for explicit validation.

## 3. Architecture Decisions

### AD-1: Complete server API surface incrementally on existing binary

**Question:** Should the server API be built incrementally or redesigned?

**Alternatives considered:**
- (A) Rewrite server from scratch with OpenAPI codegen.
- (B) Incrementally add routes to existing Chi/net-http server.
- (C) Build separate API server process.

**Chosen:** B — Incremental addition to existing server.

**Rationale:** The server already works (`cli/cmd/server.go`,
`cli/internal/server/`), serves MCP endpoints, and has the right architecture
(single binary, three surfaces per architecture doc). Adding routes is lower
risk than rewriting.

### AD-2: Web UI as separate Vite build embedded at release time

**Question:** How to integrate the web UI with the Go server?

**Chosen:** Per ADR-002 — Vite + React + Tailwind built separately, embedded via
Go `embed.FS`. Dev workflow uses Vite dev server with proxy to Go backend.

**Rationale:** Already decided and documented. Client-side data layer uses
plain JS arrays + MiniSearch per ADR-005 v2, capping bead UI at ~10,000 beads before server-side
pagination needed.

### AD-3: Registry uses shallow clone for package fetch

**Question:** How to fetch packages from git registries?

**Alternatives considered:**
- (A) GitHub API for file-by-file download.
- (B) Git archive.
- (C) Shallow clone to temp dir, copy to target, record in lockfile.

**Chosen:** C — Shallow clone.

**Rationale:** Works with any git host, enables tree hash computation from
actual git objects, and handles both single-file resources and multi-file
packages. ADR-003 defines the integrity model.

### AD-4: Skills are plain markdown files managed by registry

**Question:** How to deliver and update agent skills?

**Chosen:** SKILL.md files installed via `ddx install ddx-skills` or `ddx init`.

**Rationale:** Follows FEAT-011 spec. Skills are documents — DDx's core
competency. No compiled code required; graceful degradation if DDx not
installed.

### AD-5: Git awareness extends existing internal/git package

**Question:** How to implement auto-commit and history features?

**Chosen:** Extend existing `cli/internal/git` package which already has
autocommit support (`git.go`, `autocommit.go`).

**Rationale:** Avoids new dependency (go-git). Shells out to git binary, which
is always available in development environments.

### AD-6: Multi-agent coordination extends existing bead claims

**Question:** How to make beads safe for multi-agent use?

**Chosen:** Extend existing claim fields with machine identifier and session ID.

**Rationale:** FEAT-013 explicitly states "primitives, not orchestration."
Extending claim metadata in the existing bead envelope is the minimal change
that satisfies the requirement.

### AD-7: Real-time updates out of scope for v1

**Question:** Should the web UI receive live updates via WebSocket/SSE?

**Chosen:** Polling only for v1. TanStack Query handles cache invalidation.

**Rationale:** Simplicity. WebSocket adds complexity to the single-binary
server model. Polling is sufficient for local-first single-user default
deployment.

### AD-8: Pagination from day 1 with offset/limit

**Question:** Should server endpoints support pagination immediately?

**Chosen:** Yes. All list endpoints accept `limit` (default 100, max 1000) and
`offset` parameters. Response meta includes `total` count.

**Rationale:** Prevents API redesign when repos grow. Plain JS arrays + MiniSearch per
ADR-005 handles bead-specific filtering without server-side search.

## 4. Interface Contracts

### 4.1 HTTP API Response Envelope

All endpoints return a standard envelope:

```json
{
  "data": "...",
  "meta": {
    "generated_at": "RFC3339",
    "request_id": "uuid",
    "path": "/api/...",
    "total": 42
  }
}
```

Error envelope:

```json
{
  "error": {
    "code": "ERR_INVALID_REQUEST",
    "message": "human-readable description",
    "details": {"field": "..."}
  },
  "meta": {"request_id": "uuid", "path": "/api/..."}
}
```

HTTP status codes: 400 (malformed), 404 (not found), 409 (conflict), 422
(validation), 500 (internal).

### 4.2 REST Endpoints

**Documents:**
- `GET /api/documents` — query: type, q, updated_after, root, limit, offset; sort: updated_at desc
- `GET /api/documents/:path` — URL-encoded relative path; path traversal blocked
- `GET /api/search?q=` — full-text + frontmatter search
- `GET /api/personas/:role` — resolve persona binding from config
- `PUT /api/docs/:id` — write document content; auto-commit if FEAT-012 enabled

**Beads:**
- `GET /api/beads` — query: status, label, owner, limit, offset; sort: priority asc, created_at desc
- `GET /api/beads/:id` — single bead with full evidence history
- `GET /api/beads/ready` — ready queue (open + all deps closed)
- `GET /api/beads/blocked` — blocked list (open + at least one dep open)
- `GET /api/beads/status` — status summary counts
- `GET /api/beads/dep/tree/:id` — dependency tree
- `POST /api/beads` — create bead (JSON body)
- `PATCH /api/beads/:id` — partial update (JSON merge patch)
- `POST /api/beads/:id/claim` — claim bead (optional assignee in body)
- `POST /api/beads/:id/unclaim` — unclaim bead

**Doc Graph:**
- `GET /api/docs/graph` — full graph as JSON
- `GET /api/docs/stale` — stale documents with reasons
- `GET /api/docs/:id` — document metadata with hash + stale reasons + dependents
- `GET /api/docs/:id/deps` — dependencies
- `GET /api/docs/:id/dependents` — dependents

**Agent Sessions:**
- `GET /api/agent/sessions` — query: limit, harness, status, since; sort: timestamp desc
- `GET /api/agent/sessions/:id` — session detail with attachment references

**Executions:**
- `GET /api/exec/definitions` — query: artifact_id, limit, offset
- `GET /api/exec/definitions/:id` — definition detail
- `GET /api/exec/runs` — query: definition_id, artifact_id, status, limit, offset
- `GET /api/exec/runs/:id` — run detail with logs and result

**Registry:**
- `GET /api/registry/search?q=` — search registry index
- `GET /api/registry/installed` — installed packages
- `GET /api/registry/verify` — integrity check result
- `POST /api/registry/install` — install package (JSON body with name)
- `POST /api/registry/uninstall` — uninstall package (JSON body with name)

**Health:**
- `GET /api/health` — liveness (200 OK)
- `HEAD /api/health` — liveness (for load balancers)
- `GET /api/ready` — dependency checks (bead store, graph, harnesses)

### 5.3 Data Integrity Constraints

- Bead IDs globally unique per collection. Graph nodes require non-empty `id`.
- Lockfile must be version controlled and reviewed on drift.
- All bead-backed collections use atomic temp-file-swap writes with directory
  locks (per ADR-004).
- Agent session attachments published before metadata row (crash-safe ordering).
- Config schema rejects unknown keys in known sections and suggests supported
  keys.

### 5.4 Migration Strategy

- Config changes are additive: new sections ignored by older DDx versions.
- Registry lockfile is new; no migration from prior state.
- Legacy `.ddx/exec/` and `.ddx/metrics/` layouts remain readable as fallback
  per exec model reconciliation plan.
- `ddx doc migrate` converts `dun:` frontmatter to `ddx:` (FEAT-007).

## 6. Error Handling Strategy

### 6.1 HTTP Errors

| Code | Meaning | Example |
|------|---------|---------|
| 400 | Malformed request | Bad JSON, invalid query params |
| 404 | Resource not found | Unknown bead ID, document path, session ID |
| 409 | Conflict | Concurrent claim, install in progress |
| 422 | Validation failure | Schema errors, missing required fields |
| 500 | Internal error | File I/O failure, parse error |

All errors include stable error codes (e.g., `ERR_BEAD_NOT_FOUND`) and
actionable hints.

### 6.2 CLI Errors

- Exit 1 with actionable error message for all failures.
- `ddx doctor` surfaces all config and dependency issues.
- Structured error output with `--json` flag.

### 6.3 Registry-Specific Error Handling

- **Network unreachable:** Use cached registry index if available; warn user
  with cache age. Fail with clear message if no cache.
- **Integrity mismatch:** Fail loudly: "expected tree hash X, got Y for package
  Z from repo R at commit C". Suggest `ddx install --force` to re-fetch.
- **Package not found:** Suggest similar names from registry index.
- **Concurrent install:** Install lock prevents races; timeout with message
  suggesting `rm .ddx/install.lock/` if stale.
- **Partial install failure:** Clean up temp directory; do not update lockfile.
  Existing installation remains intact.
- **Corrupted lockfile:** `ddx verify` reports specific YAML parse error.
  `ddx install --force` recreates from fresh state.

### 6.4 Concurrency

- Bead operations: directory lock with configurable timeout; clear error on
  contention with PID of holding process.
- Collection writes: atomic temp-file-swap pattern (lock → snapshot → temp →
  atomic rename).
- Server concurrent requests: each request gets independent read snapshot.
- Stale lock cleanup: directory locks include PID; `ddx doctor` detects stale
  locks from dead processes.

### 6.5 Git Awareness Error Handling (FEAT-012)

- **Not in git repo:** Features disabled gracefully; commands return clear
  "not a git repository" error.
- **Dirty working tree:** Auto-commit only touches DDx-managed files.
- **Concurrent auto-commits (FEAT-013):** Exclusive commit lock prevents
  interleaving from multiple agents.

### 6.6 Web UI Error Handling

- API failures: TanStack Query retries with exponential backoff; error toast
  with retry option for mutations.
- Stale data: cache invalidation on mutations; stale indicator when polling
  detects changes.
- Offline: show cached data with staleness banner.
- MiniSearch index rebuild: re-index from in-memory array on data reload; client-side state is ephemeral.

### 6.7 Doc Graph Error Handling

- **Malformed frontmatter:** Warning collection + continue; exclude invalid
  docs from graph. `ddx doc validate` emits all warnings.
- **Cycles:** Explicit cycle detection with error naming the participating nodes.
- **Missing dependency target:** Fail fast with actionable path and expected
  dependency ID.

## 7. Security Considerations

### 7.1 Attack Surfaces

1. **Path traversal** via `/api/documents/:path` — URL-encoded paths must be
   canonicalized and confined to project root.
2. **Server exposure** — localhost-only default; API key required for
   non-localhost bind.
3. **Registry supply chain** — commit SHA pinning + tree hash per ADR-003.
4. **Session log leakage** — token values in agent sessions; redaction opt-in
   before persistence.
5. **Agent dispatch** — permission profiles (safe/supervised/unrestricted) gate
   capabilities.
6. **MCP write tools** — bead mutations require localhost or API key.
7. **CORS** — restricted to configured origins only.
8. **CSRF** — SameSite cookie + custom header for non-GET browser requests.
9. **XSS in web UI** — sanitize markdown HTML rendering via DOMPurify or
   equivalent before DOM insertion.
10. **SSRF via registry URLs** — restrict to HTTPS URLs only in registry config.
11. **Input size** — configurable `max_body_size` (default 10MB) for document
    writes.
12. **Environment variable leakage** — env vars not captured in agent session
    logs by default.

### 7.2 Mitigations

- Path sanitization and canonicalization in all file-serving handlers.
- API key middleware for non-localhost; key checked via constant-time compare.
- CORS middleware with explicit origin list; no wildcard.
- Content Security Policy: restrict scripts to self; no inline scripts.
- Never log raw tokens; redact token-like strings in structured logs.
- Lockfile tree-hash verification on every install/verify.
- Agent permissions default to safe mode; unrestricted requires explicit opt-in.

### 7.3 Security Modes

| Mode | Bind | Auth | Use Case |
|------|------|------|----------|
| Dev | localhost | None required | Default local development |
| Remote | configurable | API key for all endpoints | Team/CI access |

### 7.4 Auditability

- Record actor/request_id for all write operations.
- Store immutable event entries for package installs/verifications.
- Agent session logging preserves evidence chain (per FEAT-006).

### 11.6 Operational Runbooks

Known remediation procedures to document:

- **Malformed config:** `ddx doctor` diagnoses; fix `.ddx/config.yaml` syntax.
- **Missing library path:** Empty document list; `/api/health` still available.
- **Lockfile mismatch:** `ddx verify` reports diff; `ddx install --force`
  recreates.
- **Stale lock:** Check PID; remove `.ddx/*.lock/` if process dead.
- **Harness unavailable:** `ddx agent doctor` reports; install missing harness.
- **Collection corruption:** Bead auto-repair preserves `.bak`; report specific
  malformed records.
