## Intent Summary

- **Vision**: DDx keeps documents as the durable source of intent and uses the CLI, server, tracker, and agent infrastructure to make that intent executable and reviewable.
- **Requirements**: The PRD still centers DDx on reusable local runtime services, project-state serving over HTTP and MCP, bead-backed execution control, process metrics, plugin API stability, and a trustworthy public-facing site.
- **Features / Stories**: FEAT-016 and FEAT-018 now largely align with the shipped CLI and test coverage. Live repo pressure is concentrated in FEAT-002 project scoping, FEAT-006 supervisor-contract cleanup, FEAT-019 naming drift, and tracker metadata hygiene.
- **Architecture / ADRs**: `architecture.md` still makes `ddx server` a multi-project host with project-scoped adapters, but the live server remains mostly singleton-shaped. ADR-002 correctly describes the embedded-server harness (SvelteKit built to static assets, served via Go `//go:embed`). The playwright config (`cli/internal/server/frontend/playwright.config.ts`) correctly uses `--port 18080` to run the embedded server.
- **Technical Design**: SD-016's derived-metrics model is implemented, but one summary-window rule still diverges. SD-015 and API-001 still contain unresolved contract drift around routing retention, queue ordering, worktree ownership, and result-state semantics.
- **Test Plans**: TP-020 aligns with the embedded `ddx-agent` boundary. TP-002 now documents the needed multi-project MCP/server coverage, but it still records that coverage as planned rather than live.
- **Implementation Plans**: The single-project execute-bead lane remains the blocking release lane. Multi-project server implementation work is now explicitly tracked as blocked follow-on work under `ddx-aac79489`.
