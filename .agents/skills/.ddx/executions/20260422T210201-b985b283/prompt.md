    <title>Efficacy &#39;Compare arms&#39; button is unclear and misplaced: decide whether to relocate, redesign, or remove</title>
    <description>
## Observed&#xA;&#xA;The Efficacy page has a &#34;Compare&#34; button that opens a flow for adding &#34;arms&#34; and writing a prompt. User feedback: they&#39;re not sure what to do with it. It does not match their mental model of what efficacy is (a session aggregation grouped by provider / harness / model).&#xA;&#xA;## Ground truth&#xA;&#xA;- The backing feature is `Query.comparisons` and the comparison record set (`cli/internal/server/graphql/resolver_feat008.go:211-226`). It&#39;s an A/B-style &#34;run the same prompt across N arms and compare results&#34; tool. That is a real capability — just not what a rollup/efficacy dashboard is for.&#xA;- It&#39;s colocated with Efficacy today probably because both involve harness/model performance. The colocation is creating confusion: the Efficacy page promises &#34;how are things doing&#34; and the Compare button promises &#34;design an experiment&#34; — different audiences, different tasks.&#xA;&#xA;## Decision required&#xA;&#xA;Before implementation, decide one of three:&#xA;&#xA;1. **Relocate.** Move Compare to its own top-level surface (e.g., `/experiments` or `/compare`) and cleanly separate &#34;how did things go&#34; (efficacy) from &#34;design an A/B&#34; (compare). Efficacy page loses the Compare button entirely.&#xA;2. **Redesign in place.** Keep it on the Efficacy page but recast as &#34;Compare these rows&#34; — the user selects 2+ efficacy rows and launches a comparison pre-seeded from them. Removes the cold-start prompt-entry UX; connects directly to the rollup the user is already looking at.&#xA;3. **Remove.** If comparisons are not being used by real workflows, drop the feature and the associated UI/resolver. Least code, least surface area.&#xA;&#xA;The user did not mandate a specific choice — they flagged the confusion and the mental model. Option 2 aligns best with their stated mental model (efficacy = session aggregation); option 1 is the cleanest separation of concerns; option 3 is viable only if usage data supports it.&#xA;&#xA;## Proposed direction&#xA;&#xA;Run the design decision as a short spike (≤ half a day):&#xA;&#xA;- Check whether `comparisons` is referenced by anything beyond this UI page: CLI, docs, specs, other server endpoints. Deletion cost is a function of that.&#xA;- Usage check: grep agent-logs + bead evidence for any `comparison_id` or `compare` correlation over the last 30 days. Is anyone actually running comparisons?&#xA;- Write a one-page decision note in `docs/helix/02-design/` recording the choice, the evidence that supports it, and the scope of the implementation work.&#xA;&#xA;This bead&#39;s remaining acceptance criteria are written assuming the outcome is (2) &#34;redesign in place&#34; — the most-likely-to-be-built option. They should be rewritten when the decision is made if the outcome is (1) or (3).&#xA;&#xA;## Out of scope&#xA;&#xA;- Anything the Efficacy perf/source rework (`ddx-0a33bc5f`) covers.&#xA;- Multi-run orchestration changes to how comparisons execute.
    </description>
    <acceptance>
**User story:** As a developer on the Efficacy page, I understand what Compare is for the moment I see it, and I can either select rows I already care about and compare them, or I can ignore the feature entirely if it&#39;s not for me.&#xA;&#xA;**Acceptance criteria (assume outcome (2) — &#34;redesign in place&#34;; rewrite these if (1) or (3) is chosen):**&#xA;&#xA;1. **Decision note.** One-pager in `docs/helix/02-design/` records the chosen outcome (relocate / redesign / remove), the evidence behind it (usage grep, dependency scan), and the scope of remaining work. Implementation PR links to it.&#xA;&#xA;2. **Row-selection entry point.** Efficacy table rows gain a leading checkbox column. Selecting ≥2 rows enables a &#34;Compare selected&#34; button in the page header. With 0 or 1 rows selected the button is disabled (not hidden) with a tooltip explaining why.&#xA;&#xA;3. **Pre-seeded comparison.** Clicking &#34;Compare selected&#34; launches the existing comparison flow with arms pre-populated from the selected rows&#39; `(harness, provider, model)` tuples. The prompt field is still required (comparisons need a prompt) but is the only field the user has to fill in — the arm list is done.&#xA;&#xA;4. **Removed from cold-start.** The standalone &#34;Compare&#34; button on the Efficacy page (the one that creates an empty comparison) is removed. New comparisons originate from a row selection. If users legitimately want a from-scratch comparison, that path lives on a separate `/experiments` page (out of scope for this bead unless the design note calls for it).&#xA;&#xA;5. **Empty-state copy.** When the Efficacy table is empty, the Compare affordance is hidden entirely (no point selecting zero rows).&#xA;&#xA;6. **Playwright e2e.**&#xA;   - Seeds the fixture with ≥3 efficacy rows.&#xA;   - Asserts the Compare button is disabled with 0/1 selections and enabled with 2+.&#xA;   - Selects 2 rows, clicks Compare, asserts the comparison form opens with arms pre-populated from those rows and only the prompt field is empty.&#xA;&#xA;7. **No regression to efficacy perf.** All perf targets from `ddx-0a33bc5f` still hold. Adding a checkbox column does not change the payload the resolver returns.
