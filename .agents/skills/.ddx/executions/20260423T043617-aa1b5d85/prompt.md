    <title>Efficacy page is slow and sources from the wrong place: rework as session aggregation, seed ≥1k fixture, raise DB decision if in-place insufficient</title>
    <description>
## Observed&#xA;&#xA;The Efficacy page at `/nodes/.../projects/.../efficacy` loads very slowly. User feedback: efficacy should really be &#34;an aggregation of sessions grouped by provider / harness / model&#34; — which is not what the current implementation does.&#xA;&#xA;## Ground truth (current implementation)&#xA;&#xA;`cli/internal/server/graphql/resolver_feat008.go:676-833` —&#xA;&#xA;- `EfficacyRows` builds its rollup from **bead evidence events** (`kind:routing` and `kind:cost` on every closed bead), not from sessions.&#xA;- Computation path: `store.ReadAll()` → for every closed bead, read its event file → filter for routing/cost → per-(harness, provider, model) aggregate.&#xA;- There is a cache (`efficacyMemo`), keyed by a fingerprint that itself walks every event on every closed bead to compute event count + max timestamp. So cache validation costs almost as much as cache rebuild.&#xA;- Cost shape: `O(closed_beads × events_per_bead)` on every request (cache miss) and even on cache hit the fingerprint walk happens.&#xA;&#xA;## Why this is wrong (per user feedback)&#xA;&#xA;- Efficacy is a sessions-view concept: &#34;how is harness/provider/model X doing?&#34; Sessions are the natural row. Bead events are a lossy, indirect proxy — they only exist for *closed* beads, they duplicate session state, and they force coupling between the efficacy feature and bead lifecycle.&#xA;- The bead-event path misses: open beads with in-flight sessions, sessions that didn&#39;t close a bead, benchmark/quorum sessions, non-execute-bead invocations (e.g., `ddx agent run`).&#xA;- The sessions feed is already being reworked into a sharded pointer index (`ddx-2ceb02fa`). Efficacy should read from that index — single source of truth.&#xA;&#xA;## Proposed direction&#xA;&#xA;### Part 1 — Reshape the data source&#xA;&#xA;- Rebuild `EfficacyRows` as an aggregation over `sessions.index` (the sharded index from `ddx-2ceb02fa`), not over bead events.&#xA;- Grouping key: `(harness, provider, model)` — optionally filterable by project, bead, label, time range.&#xA;- Metrics per row: attempts (count), successes (count + rate), median input/output tokens, median duration, median cost. Identical shape to today; only the source changes.&#xA;- Shard awareness: time-range queries map to a shard set (same pattern as `ddx-2ceb02fa`); the default view reads current + previous shard.&#xA;&#xA;### Part 2 — Fixture + baseline (honor user&#39;s 1,000+ record floor)&#xA;&#xA;- Perf harness target: aggregate ≥**1,000 session rows** (≥ 10 distinct `(harness, provider, model)` groups) within the budget below.&#xA;- Stretch fixture: 50,000 session rows across 24 monthly shards. Measures steady-state behavior as the index grows.&#xA;- Baseline captured in the bead notes before any optimization.&#xA;&#xA;### Part 3 — Perf targets (first pass, in-place)&#xA;&#xA;- Default Efficacy view (all-time) on a 10k-session, 2-shard fixture: p95 ≤ 150ms in-process / ≤ 400ms HTTP.&#xA;- Date-filtered view (last 30 days, 1 shard): p95 ≤ 60ms in-process / ≤ 200ms HTTP.&#xA;- If in-place optimization (scope, pushdown, memoize-per-shard, streaming aggregation) **cannot hit these numbers** on the 50k stretch fixture, **do not silently change substrate** — raise the decision with the user. Candidate backends named by the user: axon, sqlite, postgres. Bring a recommendation + rationale, not a fait accompli.&#xA;&#xA;### Part 4 — Drop the bead-event read path&#xA;&#xA;- After the rewrite, remove `efficacyFingerprint`, `buildEfficacySnapshot` event iteration, `attemptsFromEvidence`, and `efficacyMemo`. They become dead weight once the source moves.&#xA;- Evidence events on beads remain (they&#39;re bead audit trail, not efficacy data); nothing in bead lifecycle changes.&#xA;&#xA;## Out of scope&#xA;&#xA;- The `comparisons` / &#34;compare arms&#34; feature on this page (tracked separately — see companion bead).&#xA;- Project-scope filtering UI (the backend support can be added now; the UI widget is a follow-up).&#xA;- Cross-machine / federated efficacy rollups.
    </description>
