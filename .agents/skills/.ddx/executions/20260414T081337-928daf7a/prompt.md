    <title>Dashboard UI: live workers page with per-worker prompt, log, and utilization detail</title>
    <description>
The server knows everything about running workers (state, harness, model, bead, attempts, prompt file, structured event log, token/cost/duration) via /api/agent/workers endpoints but the web UI has no live workers view. When an operator selects the eitri node they should land on (or reach in one click) an Agents page that lists every running worker from /api/agent/workers with bead-id, harness, model, state, elapsed, and latest status line. Clicking a row should open a worker detail page showing: (1) the rendered bead prompt from .ddx/executions/&lt;attempt&gt;/prompt.md, (2) the structured session log stream from .ddx/agent-logs/agent-loop-&lt;session&gt;.jsonl rendered via FormatSessionLogLines (so tool calls, turn counts, and timings are readable), (3) a session utilization panel with input/output/total tokens, cost-if-available, duration, attempts and successes/failures from the worker record, and (4) the base-rev and result-rev link once the worker has made progress. Polling at ~2s is fine for v1; no WebSocket required.\n\nBacking endpoints that already exist:\n- GET /api/agent/workers -&gt; list\n- GET /api/agent/workers/:id -&gt; detail\n- GET /api/agent/workers/:id/log -&gt; stdout+stderr streams\n\nMissing backing endpoints (if not already there): GET /api/agent/workers/:id/prompt to serve prompt.md content without requiring the UI to know the execution bundle path; optionally /api/agent/workers/:id/events to serve the structured JSONL stream if /log doesn&#39;t already cover it.
    </description>
    <acceptance>
Web UI route /nodes/:nodeId/agents (or /workers) lists all workers from /api/agent/workers with auto-refresh every ~2s. A worker row links to /nodes/:nodeId/agents/:workerId with three panels: prompt, log stream (structured events rendered with FormatSessionLogLines symbols), utilization (tokens/cost/duration/attempts). Both pages work against the alpha14 server without any additional backend work beyond optionally exposing the prompt content. Playwright test covers: list-shows-running-worker, list-refreshes-on-state-change, detail-renders-prompt-log-and-utilization.
    </acceptance>
