    <title>Simplify execute-bead worker: task-only, no merge/preserve/gate logic</title>
    <description>
The current execute-bead worker does WAY too much: creates a worktree, runs the agent, evaluates required gates, decides merge-by-default vs preserve vs land-conflict, writes preserve refs, synthesizes commits on IsDirty, classifies 7+ outcomes, manages heartbeats, handles claim/unclaim contention, runs recoverOrphans on stale worktrees. This surface is why (a) two concurrent workers race on the main-branch merge and the loser&#39;s good work ends up under refs/ddx/iterations/ needing manual rescue, (b) no_changes gets misclassified as synthesized-commit success, (c) the gate-enforcement path and the merge path are entangled, and (d) the worker needs special privileges (update-ref, merge, rebase) that a simple task-runner should never have.\n\nThe correct design is the one the operator just described: execute-bead is a thin worker. &#39;I have been handed a worktree and a task. I will execute that task and make commits in the worktree. I will stream progress events to ddx server. I will notify ddx server when I am done, reporting the worktree branch and the commit(s) I made. Then I exit.&#39; That&#39;s it.\n\nEverything else — merge to main, conflict resolution, required-gate evaluation, preserve-under-ref, decide whether to close the bead — moves to the PARENT process (the operator, or a driver tool running interactively in the terminal). Merge conflicts are resolved interactively the same as any other merge. A human (or an outer agent with full git authority) reviews the agent&#39;s commits and decides what to do with them.\n\nThis simplification deletes most of ExecuteBead() in cli/internal/agent/execute_bead.go and moves:\n- resolveBase / baseRev tracking → parent\n- Merge() call → parent\n- preserveRef / UpdateRef / refs/ddx/iterations/ writes → parent\n- evaluateRequiredGates → parent (or a separate &#39;ddx agent check&#39; step)\n- outcome classification &#39;merged/preserved/land-conflict/error&#39; → parent\n- recoverOrphans → parent (cleanup runs when parent next launches workers)\n- heartbeatIsStale reclaim logic → parent (parent tracks worker health)\n\nWhat the worker keeps:\n- Take a worktree dir + task description (prompt) as input\n- Launch the agent harness with the prompt\n- Stream progress events to ddx server (session log + structured JSONL)\n- On exit, emit a final event: worktree path, branch, commit shas (or empty), exit code, duration\n- That&#39;s it\n\nExecute-loop likewise shrinks or goes away. In the clean design, a driver loop in the parent process picks ready beads, creates a /tmp worktree per bead, calls &#39;ddx agent execute-bead&#39; (worker-only mode), waits for the worker&#39;s exit event from ddx server, reviews the result, and merges or discards. When two beads are in flight concurrently, the parent serializes the merge step — no race possible.
    </description>
    <acceptance>
cli/internal/agent/execute_bead.go ExecuteBead() contains ONLY: worktree setup (or accept pre-made worktree), prompt construction, agent harness dispatch, progress event emission, and a final completion event. No Merge, no UpdateRef, no preserveRef, no evaluateRequiredGates, no outcome classification beyond {task_succeeded, task_failed, task_no_changes}. All merge/preserve/gate logic moved to a new parent-side orchestrator that can be run interactively. Regression test: two workers on two different beads complete in arbitrary order; parent process merges both sequentially with interactive conflict resolution if needed; no worktree-internal merge code paths are exercised.
    </acceptance>
