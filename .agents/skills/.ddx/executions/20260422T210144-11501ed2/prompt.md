    <title>workersByProject compares project id to project path: per-project workers view always empty</title>
    <description>
## Observed&#xA;&#xA;User clicks &#34;Drain queue&#34; on the project home page. Mutation returns success with a worker id. Navigating to `/nodes/.../projects/&lt;id&gt;/workers` shows no workers — an empty table.&#xA;&#xA;Verified on disk: the worker record exists and is running.&#xA;&#xA;```&#xA;$ cat .ddx/workers/worker-20260422T205846-f23f/status.json&#xA;{&#xA;  &#34;id&#34;: &#34;worker-20260422T205846-f23f&#34;,&#xA;  &#34;kind&#34;: &#34;execute-loop&#34;,&#xA;  &#34;state&#34;: &#34;running&#34;,&#xA;  &#34;project_root&#34;: &#34;/Users/erik/Projects/ddx&#34;,&#xA;  ...&#xA;}&#xA;```&#xA;&#xA;So the dispatch worked, but the per-project workers list filters it out.&#xA;&#xA;## Root cause&#xA;&#xA;`cli/internal/server/state_graphql.go:156` in `GetWorkersGraphQL`:&#xA;&#xA;```go&#xA;if projectID != &#34;&#34; &amp;&amp; rec.ProjectRoot != projectID {&#xA;    continue&#xA;}&#xA;```&#xA;&#xA;- `projectID` is the GraphQL argument — a project **id** like `proj-96d7ea83`.&#xA;- `rec.ProjectRoot` is the worker record field — a project **path** like `/Users/erik/Projects/ddx`.&#xA;&#xA;These are different representations and never match. With any non-empty `projectID`, every worker is filtered out.&#xA;&#xA;The global `workers` resolver (`resolver_agent.go:9`) passes `&#34;&#34;` and works correctly. Only the per-project path (`workersByProject` → `GetWorkersGraphQL(projectID)`) is broken. The user&#39;s flow (dispatch from project home → navigate to project workers) hits exactly this path.&#xA;&#xA;Note: `GetBeadSnapshots` (state_graphql.go:49-105) handles the same concept correctly by iterating `s.Projects` and filtering on `proj.ID != projectID` — comparing id to id. The worker resolver does not use that pattern.&#xA;&#xA;## Proposed direction&#xA;&#xA;Two options; prefer (1) for minimal change, (2) for long-term clarity.&#xA;&#xA;### Option 1 — Resolve id → path at the filter site&#xA;&#xA;```go&#xA;expectedPath := &#34;&#34;&#xA;if projectID != &#34;&#34; {&#xA;    proj, ok := s.GetProjectByID(projectID)&#xA;    if !ok {&#xA;        return nil // unknown project id → no workers&#xA;    }&#xA;    expectedPath = proj.Path&#xA;}&#xA;...&#xA;if expectedPath != &#34;&#34; &amp;&amp; rec.ProjectRoot != expectedPath {&#xA;    continue&#xA;}&#xA;```&#xA;&#xA;### Option 2 — Store project id on the worker record&#xA;&#xA;Add `ProjectID string` to `WorkerRecord`. Populate at creation time in `WorkerManager.StartExecuteLoop` via `s.GetProjectByPath(effectiveRoot)`. Filter on `rec.ProjectID != projectID`. Cleaner — the worker record carries its own identity and the query doesn&#39;t depend on runtime resolution. Requires a tiny migration (for older records missing the field, fall back to path-resolve as in Option 1).&#xA;&#xA;Recommend Option 2. It removes the representation mismatch for good and is cheap.&#xA;&#xA;## Scope note&#xA;&#xA;This is a one-line functional bug with outsized impact: it&#39;s the reason the Workers page looks broken whenever used as the dashboard intends. Prioritizing P1.&#xA;&#xA;## Out of scope&#xA;&#xA;- Workers pane lifecycle controls (`ddx-69789664`).&#xA;- Cross-project workers view — the global `workers` query already handles that.
