    <description>
## Observed&#xA;&#xA;The Sessions page at `/nodes/.../projects/.../sessions` shows nothing newer than **2026-04-19**, three days ago. ddx is definitely still running — `.ddx/executions/` contains bundles from 2026-04-22 (today), `agent-s-*.jsonl` from 2026-04-21, `agent-loop-*.jsonl` from 2026-04-20. The UI simply cannot see them.&#xA;&#xA;## Root cause&#xA;&#xA;The Sessions GraphQL resolver reads exclusively from `.ddx/agent-logs/sessions.jsonl` (`cli/internal/server/state_graphql.go:504-516`). That file stopped being appended on 2026-04-19 19:46.&#xA;&#xA;The only writer is `Runner.logSession` (`cli/internal/agent/runner.go:847-914`). It is called from four sites today:&#xA;&#xA;- `cli/internal/agent/runner.go:229` (Runner.Run — legacy)&#xA;- `cli/internal/agent/agent_runner_service.go:216`&#xA;- `cli/internal/agent/claude_stream.go:516`&#xA;- `cli/internal/agent/virtual.go:206`&#xA;&#xA;Production execute-bead / execute-loop no longer goes through any of them. The Runner-to-service refactor series (`5b933103`, `280f58e0`, `74eceda9`, `7cc1eb58`, `e05c785e`, `9f79f037`) routed the common path through `service.Execute` directly and did not carry forward the sessions.jsonl append. The last sessions.jsonl write lines up with the day that refactor set became default.&#xA;&#xA;## Secondary issue (same fix should address)&#xA;&#xA;`sessions.jsonl` is 397 MB for only 895 entries because each entry embeds the full `result.Output` (see `SessionEntry.Response` at `runner.go:879`). Growth is unbounded, there is no rotation, and `GetAgentSessionsGraphQL` (`state_graphql.go:476-490`) reads the whole file, parses every entry, and sorts in memory on every query. Even if we just restore the writer, the feed is fragile and will become a perf problem again quickly — relevant to `ddx-ad0db8fd` (bead detail was also O(N) scan).&#xA;&#xA;## Proposed direction&#xA;&#xA;Do not simply paper over by re-plumbing `logSession` into `service.Execute`. The right fix is:&#xA;&#xA;1. **Decide the source of truth for &#34;a session&#34;.** Current candidates on disk:&#xA;   - `.ddx/executions/&lt;ts&gt;-&lt;hash&gt;/{manifest.json,prompt.md,result.json}` — already written per execute-bead attempt.&#xA;   - `.ddx/agent-logs/agent-*.jsonl` and `agent-s-*.jsonl` — per-run streams from the agent harness.&#xA;   - Bead evidence events (kind:cost already written per attempt, see `execute_bead.go:492`).&#xA;   A session record should be derivable from a combination of execution bundles + agent-log streams, not from a separately-appended aggregate. The aggregate should be an **index**, not a **copy**.&#xA;&#xA;2. **Introduce a lightweight index.** A thin `.ddx/agent-logs/sessions.index.jsonl` containing `{id, projectID, beadID, harness, model, startedAt, endedAt, durationMs, cost, tokens, outcome, bundlePath}` — pointer fields only, no embedded response bodies. Existing `sessions.jsonl` is deprecated to a read-only legacy file during a grace period.&#xA;&#xA;3. **Write the index from `service.Execute`** (or a wrapper the execute-bead / run / execute-loop paths all share). One place to touch. Unit test guards that every production code path that executes an agent appends exactly one index row.&#xA;&#xA;4. **GraphQL reads the index, streams bodies on demand.** `sessions { edges { node { ... } } }` reads the index. A new `session(id: ID!) { prompt, response, stderr }` lazy-loads the heavy fields from the execution bundle or agent-log file when the user expands a row.&#xA;&#xA;5. **Legacy sessions.jsonl compatibility.** The index can be seeded once from the existing 397 MB file so historical data is not lost, then the old file is not read again. One-shot migration command (`ddx agent log reindex` or similar).&#xA;&#xA;## Out of scope&#xA;&#xA;- Replaying or regenerating sessions for the 2026-04-19 → 2026-04-22 gap. If the evidence exists in executions/agent-logs we can backfill the index; if not, accept the gap and cap it with a dated marker in the UI.&#xA;- A full telemetry rework. This bead just restores visibility and fixes the shape.

## 2026-04-22 — User-directed amendment: shard the index&#xA;&#xA;Per user feedback: a single `sessions.index.jsonl` just replays the same problem at smaller scale. The aggregate needs to be split.&#xA;&#xA;### Revised storage shape&#xA;&#xA;- Directory: `.ddx/agent-logs/sessions/` (per-project, under each registered project&#39;s `.ddx`).&#xA;- Inside: **monthly shards** named `sessions-YYYY-MM.jsonl`.&#xA;- Each shard is pointer-only, same fields as the previously-proposed index (id, projectID, beadID, harness, surface, model, startedAt, endedAt, durationMs, cost, tokens, outcome, exitCode, bundlePath, nativeLogRef).&#xA;- Writer resolves the target shard from `time.Now().UTC()` at append time and creates the shard file lazily on first write of the month.&#xA;- Heavy bodies (prompt, response, stderr) stay where they already live — `.ddx/executions/&lt;ts&gt;-&lt;hash&gt;/` and agent-log streams — and are lazy-loaded by the `session(id:)` resolver.&#xA;&#xA;### Why monthly, not daily / size-based / SQLite&#xA;&#xA;- **Daily** — produces tens of thousands of files over the project lifetime; ReadDir becomes the cost.&#xA;- **Size-based** — requires coordinating rotation with the append; race-prone and hard to test.&#xA;- **SQLite** — overkill for first pass; revisit if a single monthly shard grows past tens of MB.&#xA;- **Monthly** — ≤12 shards/year, human-readable filenames, date-range queries map trivially to shard selection, bounded shard size in practice.&#xA;&#xA;### Reader behavior&#xA;&#xA;- Default Sessions query loads the **current + previous shard** (covers &#34;recent activity&#34; for any current date).&#xA;- Date-range queries map the range to a shard set and read only those.&#xA;- Pagination beyond the default window triggers older-shard loads on demand.&#xA;- No shard-coordination lock needed: each month&#39;s file is append-only and only the current-month shard is actively written.&#xA;&#xA;### Legacy migration&#xA;&#xA;- `ddx agent log reindex` reads the existing `.ddx/agent-logs/sessions.jsonl` once and splits by each entry&#39;s `timestamp` field into the appropriate monthly shards under `.ddx/agent-logs/sessions/`.&#xA;- After migration, the legacy file is renamed `sessions.jsonl.legacy` (not deleted) and never read again by the server. Hand-removal is the user&#39;s call.&#xA;- Idempotent: re-running the command merges without duplicating (index row dedup by `id`).&#xA;&#xA;### Revised acceptance criteria (supersedes criteria 1, 5, 6, 7 on original description)&#xA;&#xA;1. **Sharded index writer.** Single shared call site (from `service.Execute` or equivalent wrapper) appends one line to `.ddx/agent-logs/sessions/sessions-YYYY-MM.jsonl`, computing the shard from `time.Now().UTC()`. Test: write across a simulated month boundary and assert two shards exist with the correct split.&#xA;&#xA;5. **Legacy migration shards too.** `ddx agent log reindex` splits `sessions.jsonl` into monthly shards by entry timestamp. Integration test uses a fixture with entries across 3 distinct months and asserts 3 shards are produced with correct contents and no duplicates on re-run.&#xA;&#xA;6. **Server reads the shard directory.** `GetAgentSessionsGraphQL` reads `.ddx/agent-logs/sessions/*.jsonl`, respects any date-range arg, and loads only the shards that intersect the range. Unit test seeds 6 monthly shards and asserts a 2-month query reads exactly 2 files.&#xA;&#xA;7. **Perf on sharded index.** Default Sessions view (current + previous shard) resolves in ≤50ms p95 on a fixture with 12 monthly shards × 10k entries each (120k total). Capture baseline (single-file read of same corpus) vs sharded in bead notes.&#xA;&#xA;(Criteria 2, 3, 4, 8, 9 from the original acceptance list remain as written.)&#xA;&#xA;### Scope clarifier&#xA;&#xA;This amendment does not change the single-call-site writer guarantee or the lazy-body resolver — those remain the load-bearing ideas. The amendment is specifically about **how the pointer index is physically stored**.&#xA;## 2026-04-23 — Implementation benchmark and evidence&#xA;&#xA;Baseline: `go test ./internal/agent -run &#39;^$&#39; -bench &#39;Benchmark(ReadSessionIndexDefaultWindow|LegacySessionsJSONLFullScan)$&#39; -benchtime=1x` measured legacy single-file full scan/sort of 120k legacy-shaped rows with 2KB embedded responses at `BenchmarkLegacySessionsJSONLFullScan-12 1 1731705957 ns/op`.&#xA;&#xA;After fix: the same command measured the sharded default-window reader at `BenchmarkReadSessionIndexDefaultWindow-12 1 8103661 ns/op` for 12 monthly shards x 10k rows each (120k total), with the default query reading only the current + previous shard window.
