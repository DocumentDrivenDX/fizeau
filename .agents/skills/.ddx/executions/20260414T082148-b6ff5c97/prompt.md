    <title>no_changes reports must include a rationale the operator can act on</title>
    <description>
Today when execute-bead exits with no_changes, the operator sees &#39;agent made no commits&#39; and nothing else. That&#39;s useless for triage — the operator cannot tell whether:\n- the bead is genuinely already satisfied (work present in a prior commit)\n- the agent gave up because it couldn&#39;t find the relevant files\n- the agent classified the task as ambiguous or too risky\n- the agent hit a tool limit or context cap\n- the prompt contract was unclear\n\nThe fix is to make no_changes a structured result that carries a short rationale from the agent itself, written at the point where the agent decided not to commit. Example of what the operator should see:\n\n  status:    no_changes\n  rationale: The epic filter requested by this bead was already added in\n             commit 1da6495 (cli/internal/bead/store.go readyFiltered). I\n             reviewed the current code and tests, confirmed the filter\n             excludes &#39;epic&#39; beads from ReadyExecution, and verified\n             TestReadyExecutionExcludesEpics exists and passes. Nothing\n             to do. Ref: 1da6495, TestReadyExecutionExcludesEpics.\n\nThis is specifically the case that motivated this bead: a continuous-drain worker processed ddx-b44ca394 (&#39;Prevent execute-loop from launching open epics by default&#39;) after the epic filter had already been implemented on main. The agent instead made a synthesized no-op commit that only retargeted 17 .claude/skills/helix-* symlinks, which the loop classified as &#39;success&#39; because a commit was produced. Had no_changes carried a rationale pointing at the prior commit, the loop could have closed the bead as &#39;already_satisfied&#39; and moved on cleanly.\n\nScope:\n1. Execute-bead prompt contract must explicitly instruct the agent: if you determine the bead&#39;s work is already present or cannot be done, report that in a structured way — do NOT make a synthesized commit to &#39;land something&#39;. Today the prompt says &#39;Making no commits (no_changes) should be rare&#39; which actively pushes agents toward the wrong behavior.\n2. The agent&#39;s exit mechanism needs a way to carry a rationale string back. Simplest version: the agent writes a file to the worktree like .ddx/executions/&lt;attempt-id&gt;/no_changes_rationale.txt before exiting. ExecuteBead reads it if no commits were produced and attaches the text to the result.NoChangesRationale field.\n3. The ExecuteBeadResult struct gains a NoChangesRationale string field, populated only when outcome == no_changes. Status: success and the unused &#39;reason&#39; field need not change.\n4. execute-loop close semantics: when no_changes arrives with a non-empty rationale pointing at a prior commit, the loop can optionally close the bead as already_satisfied with the rationale recorded in the close event body (instead of waiting for MaxNoChangesBeforeClose=3 strikes). Gate this behind a &#39;rationale references existing commit&#39; heuristic to avoid flapping — if the rationale is a vague &#39;nothing to do&#39;, keep current cooldown behavior.\n5. ddx server worker log should render the rationale prominently, not bury it in session metadata.\n6. The prompt contract note that no_changes should be &#39;rare&#39; is replaced with: &#39;If the work is already present, say so and provide a reference (commit SHA, test name, file path). A well-justified no_changes is always preferred to a cosmetic synthesized commit.&#39;
    </description>
